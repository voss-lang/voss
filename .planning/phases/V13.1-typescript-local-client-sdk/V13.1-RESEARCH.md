## Summary

This phase produces two parallel artifacts: (1) the shared contract snapshot (`contracts/openapi.json` + event-union schema) with a Python exporter and a drift gate that the other SDK sub-phases (V13.2–.4) consume, and (2) the `@vosslang/sdk` TypeScript package at `sdk/typescript/` with an isomorphic core (REST + SSE + permission helpers + generated types) and a Node-only launcher entrypoint.

The Python server surface is fully in place and well-tested. The `_force_event_schema` hook in `app.py` already publishes `EventEnvelope` into the OpenAPI components, meaning a single `openapi-typescript` pass over `contracts/openapi.json` yields BOTH REST types and the full 21-member `AgentEvent` tagged union — no extra schema wiring is needed. The only infrastructure that must be created in Wave 0 is the `contracts/` directory and the exporter script; everything else is additive SDK code.

The chosen stack (openapi-typescript 7.13, openapi-fetch 0.17, eventsource-parser 3.1, tsup 8.5, vitest 4.1, dependency-cruiser 17.4) is current and legitimate. All packages have GitHub source repos, multi-year registry history, and no postinstall scripts.

**Primary recommendation:** Follow the wave sequence: (1) produce the Python contract snapshot + drift gate; (2) run `openapi-typescript` and check in types; (3) implement the browser-safe core against the generated types; (4) add the Node launcher behind `./node`; (5) wire integration tests with `VOSS_SERVE_FAKE_TURN`.

### Pattern 1: Static OpenAPI Export (Python Exporter Script)

**What:** A Python script that calls `create_app(<fixed-token>).openapi()` and `EventEnvelope.model_json_schema()`, writing two committed JSON files. Uses a fixed token so output is byte-stable across runs.

**When to use:** Wave 0, before any TS codegen. Also re-run by the CI drift gate.

**Exporter shape (`scripts/export_contract.py`):**
```python
# Source: derived from app.py:create_app + events.py:EventEnvelope (repo verified)
import json
from pathlib import Path

FIXED_TOKEN = "voss-contract-export-fixed-token-v1"
OUT = Path(__file__).resolve().parents[1] / "contracts"

from voss.harness.server.app import create_app
from voss.harness.server.events import EventEnvelope

def main():
    OUT.mkdir(exist_ok=True)
    # REST + event-union OpenAPI schema (EventEnvelope forced in by _force_event_schema)
    schema = create_app(FIXED_TOKEN).openapi()
    (OUT / "openapi.json").write_text(
        json.dumps(schema, indent=2, sort_keys=True) + "\n"
    )
    # Standalone event-union JSON Schema (for V13.2+ typify/serde consumers)
    event_schema = EventEnvelope.model_json_schema(
        ref_template="#/components/schemas/{model}"
    )
    (OUT / "events.schema.json").write_text(
        json.dumps(event_schema, indent=2, sort_keys=True) + "\n"
    )
    print(f"Exported to {OUT}/")

if __name__ == "__main__":
    main()
```

**Why `sort_keys=True`:** Pydantic 2's `.model_json_schema()` output order is deterministic for a given Python version, but `sort_keys=True` provides an extra byte-stability guarantee across minor Pydantic releases that might reorder `$defs`.

**Important:** `create_app()` calls `_force_event_schema(app)` internally, so the schema already includes `EventEnvelope` and all 21 member schemas in `components/schemas`. No separate wiring needed.

### Pattern 2: Python Drift Gate Test

**What:** A pytest test that re-runs the exporter and compares against the committed snapshot. Mirrors the codegen snapshot pattern in `tests/codegen/test_snapshots.py`.

**Pattern mirrors:** `tests/codegen/test_snapshots.py` → reads committed snapshot, re-generates, asserts byte-identical. [CITED: tests/codegen/test_snapshots.py — verified in repo]

```python
# Source: mirrors tests/codegen/test_snapshots.py pattern (repo verified)
# tests/harness/server/test_contract_drift.py
import json
from pathlib import Path

import pytest

CONTRACTS = Path(__file__).resolve().parents[3] / "contracts"

def _regenerate():
    from voss.harness.server.app import create_app
    from voss.harness.server.events import EventEnvelope
    FIXED_TOKEN = "voss-contract-export-fixed-token-v1"
    schema = create_app(FIXED_TOKEN).openapi()
    openapi_json = json.dumps(schema, indent=2, sort_keys=True) + "\n"
    event_schema = EventEnvelope.model_json_schema(
        ref_template="#/components/schemas/{model}"
    )
    events_json = json.dumps(event_schema, indent=2, sort_keys=True) + "\n"
    return openapi_json, events_json

def test_openapi_snapshot_not_drifted():
    """Fail if the live app.openapi() differs from the committed contracts/openapi.json."""
    committed = (CONTRACTS / "openapi.json").read_text()
    regenerated, _ = _regenerate()
    assert regenerated == committed, (
        "contracts/openapi.json is stale — re-run scripts/export_contract.py and commit"
    )

def test_events_schema_not_drifted():
    """Fail if the live EventEnvelope schema differs from committed contracts/events.schema.json."""
    committed = (CONTRACTS / "events.schema.json").read_text()
    _, regenerated = _regenerate()
    assert regenerated == committed, (
        "contracts/events.schema.json is stale — re-run scripts/export_contract.py and commit"
    )

def test_drift_gate_detects_synthetic_drift(tmp_path):
    """Verify the gate actually fails when a field is added without re-snapshotting."""
    # Corrupt the committed copy in a tmp location and check it doesn't equal live
    openapi_json, _ = _regenerate()
    corrupted = openapi_json.replace('"title":', '"title_x":')
    assert corrupted != openapi_json
```

### Pattern 3: openapi-typescript Codegen (CLI)

**What:** CLI invocation to generate TS types from the committed `contracts/openapi.json`. Current major version is 7 (7.13.0). [VERIFIED: npm registry]

**CLI command:**
```bash
npx openapi-typescript contracts/openapi.json -o sdk/typescript/src/generated/types.ts
```

**Key behavior for discriminated unions:** When Pydantic 2 emits a discriminated union with `oneOf` + `discriminator`, openapi-typescript 7.x renders each member as its own interface and the union as a TypeScript `|` union. The `discriminator.propertyName` field does NOT automatically produce a TS discriminated union with string literal narrowing — the discriminator is preserved as a per-member `type` literal field. [CITED: openapi-ts.dev/advanced]

**The key insight:** Each `AgentEvent` member already has `type: Literal["<name>"]` as its model field. Pydantic 2's `model_json_schema()` emits these as `{ "type": { "const": "<name>" } }` in each member schema. openapi-typescript 7 renders `const` as a TypeScript string literal type. So the generated union is:

```typescript
// Source: openapi-ts.dev/advanced (discriminated union rendering)
export type AgentEvent =
  | { type: "server.connected"; v: number }
  | { type: "session.idle"; v: number; session_id: string }
  | { type: "stream.delta"; v: number; text: string }
  | /* ... 21 members ... */;
```

This IS a TypeScript discriminated union — `switch (event.type)` narrows exhaustively when combined with `satisfies` or `never` checks. [CITED: openapi-ts.dev/advanced]

**Watch for:** Pydantic 2's `Field(discriminator="type")` annotated union emits an `oneOf` in the top-level `EventEnvelope` component, with each member having a `const` for the `type` field. Verify the generated types file that `type` is rendered as a string literal (not `string`) for each member.

### Pattern 4: openapi-fetch Client with Bearer Auth

**What:** Create a typed REST client from the generated `paths` type, inject `Authorization: Bearer` on every request via middleware. [CITED: openapi-ts.dev/openapi-fetch/middleware-auth]

```typescript
// Source: openapi-ts.dev/openapi-fetch (verified via Context7)
import createClient, { type Middleware } from "openapi-fetch";
import type { paths } from "./generated/types";

export function createVossClient(baseUrl: string, token: string) {
  const client = createClient<paths>({ baseUrl });
  
  const authMiddleware: Middleware = {
    async onRequest({ request }) {
      request.headers.set("Authorization", `Bearer ${token}`);
      return request;
    },
  };
  
  client.use(authMiddleware);
  return client;
}
```

**Error surfacing:** `openapi-fetch` returns `{ data, error }` — on a non-2xx response, `data` is undefined and `error` contains the response body. The planner should define a typed error class: `class VossApiError extends Error { status: number; detail: string }`. Throw it from a response middleware hook.

### Pattern 5: SSE Client with eventsource-parser (isomorphic)

**What:** Open `GET /session/:id/events` with Bearer header, parse SSE framing via `EventSourceParserStream`, yield typed `AgentEvent` values, abort-cancel on consumer teardown. [CITED: github.com/rexxars/eventsource-parser README]

```typescript
// Source: eventsource-parser README (verified via Context7)
import { EventSourceParserStream } from "eventsource-parser/stream";
import type { AgentEvent } from "./generated/types";

export async function* subscribeToEvents(
  baseUrl: string,
  sessionId: string,
  token: string,
  signal?: AbortSignal,
): AsyncIterable<AgentEvent> {
  const response = await fetch(
    `${baseUrl}/session/${sessionId}/events`,
    {
      headers: { Authorization: `Bearer ${token}` },
      signal,
    }
  );

  if (!response.ok || !response.body) {
    throw new VossApiError(response.status, await response.json());
  }

  const stream = response.body
    .pipeThrough(new TextDecoderStream())
    .pipeThrough(new EventSourceParserStream());

  for await (const event of stream) {
    if (event.data) {
      const parsed = JSON.parse(event.data) as AgentEvent;
      yield parsed;
    }
  }
}
```

**Abort behavior:** Passing `signal` from an `AbortController` to `fetch` cancels the in-flight request. The server-side SSE generator's `CancelledError` then cancels the turn task (per `app.py:408-416`). Per PROTOCOL §8, this is the correct abort path — no separate `POST /abort` needed for consumer-driven teardown.

**Voss wire framing detail:** The server uses `EventSourceResponse` from `sse-starlette`. Wire format is:
```
event: <type>\n
data: <json>\n
id: <seq>\n
\n
```
`eventsource-parser` handles this correctly. The `event.name` field from the parser matches the SSE `event:` line (e.g., `"stream.delta"`). The `event.data` field is the JSON payload.

### Pattern 6: Node Launcher / Supervisor

**What:** Node-only module that spawns `voss serve`, reads one-line handshake from stdout, exposes a configured client, and kills the child on dispose. [CITED: voss/harness/server/serve.py — verified in repo; tests/harness/mcp/test_mcp_serve_e2e.py — verified subprocess pattern]

**Exact spawn command:** `python -m voss.cli serve` (confirmed from `pyproject.toml` entry point `voss = "voss.cli:main"` and `serve_cmd` registered at cli.py:4183). The npm shim at `npm/bin/voss.js` (from `@vosslang/cli`) can also be used if Python is vendored via the npm package, but the direct Python invocation is more reliable in CI.

**Critical details from serve.py:**
1. `sock.listen(128)` runs BEFORE the handshake print — race-free; the OS queues connections.
2. Handshake is exactly one JSON line: `{"v":1,"port":<N>,"token":"<str>"}\n`.
3. Server self-terminates when stdin reaches EOF (`_watch_stdin_eof` daemon thread).
4. Server also polls `getppid()` every 2s as a secondary orphan guard.

```typescript
// Source: mirrors mcp/test_mcp_serve_e2e.py _spawn_server pattern + serve.py handshake
// sdk/typescript/src/launcher/launcher.ts
import { spawn, type ChildProcess } from "node:child_process";
import { createInterface } from "node:readline";

export interface VossHandshake {
  v: number;
  port: number;
  token: string;
}

export class VossLauncher {
  private proc: ChildProcess | null = null;

  async start(options: { python?: string; cwd?: string } = {}): Promise<{ client: VossClient; pid: number }> {
    const python = options.python ?? "python3";
    this.proc = spawn(python, ["-m", "voss.cli", "serve"], {
      cwd: options.cwd ?? process.cwd(),
      stdio: ["pipe", "pipe", "inherit"],
      env: process.env,
    });

    const handshake = await this._readHandshake();
    const baseUrl = `http://127.0.0.1:${handshake.port}`;
    const client = createVossClient(baseUrl, handshake.token);

    // Kill-on-drop: ensure no orphan on process exit
    process.on("exit", () => this.dispose());

    return { client, pid: this.proc.pid! };
  }

  private _readHandshake(): Promise<VossHandshake> {
    return new Promise((resolve, reject) => {
      const rl = createInterface({ input: this.proc!.stdout! });
      const timer = setTimeout(() => reject(new Error("handshake timeout")), 10_000);
      rl.once("line", (line) => {
        clearTimeout(timer);
        rl.close();
        resolve(JSON.parse(line));
      });
      this.proc!.once("error", reject);
      this.proc!.once("exit", (code) => reject(new Error(`serve exited early: ${code}`)));
    });
  }

  dispose(): void {
    if (this.proc) {
      this.proc.stdin?.end();  // EOF → server's _watch_stdin_eof fires
      this.proc.kill("SIGTERM");
      this.proc = null;
    }
  }
}
```

### Pattern 9: vitest Integration Test — spawning `voss serve`

**What:** vitest test file that spawns `voss serve` as a child process (mirroring the MCP e2e test pattern), reads the handshake, drives a turn via `VOSS_SERVE_FAKE_TURN`, and tears down cleanly. [CITED: tests/harness/mcp/test_mcp_serve_e2e.py — verified in repo]

```typescript
// Source: mirrors tests/harness/mcp/test_mcp_serve_e2e.py pattern
// sdk/typescript/tests/helpers/serve-fixture.ts
import { spawn, type ChildProcess } from "node:child_process";
import { createInterface } from "node:readline";

export interface ServeFixture {
  baseUrl: string;
  token: string;
  pid: number;
  proc: ChildProcess;
  teardown(): void;
}

export async function spawnVossServe(cwd: string): Promise<ServeFixture> {
  const proc = spawn(
    process.env.PYTHON_BIN ?? "python3",
    ["-m", "voss.cli", "serve"],
    {
      cwd,
      stdio: ["pipe", "pipe", "inherit"],
      env: {
        ...process.env,
        VOSS_SERVE_FAKE_TURN: "1",  // <- CI seam: no provider creds needed
      },
    }
  );

  const handshake: { v: number; port: number; token: string } = await new Promise(
    (resolve, reject) => {
      const rl = createInterface({ input: proc.stdout! });
      const timer = setTimeout(() => reject(new Error("handshake timeout")), 10_000);
      rl.once("line", (line) => {
        clearTimeout(timer);
        rl.close();
        resolve(JSON.parse(line));
      });
      proc.once("error", reject);
      proc.once("exit", (code) => reject(new Error(`serve exited early: ${code}`)));
    }
  );

  return {
    baseUrl: `http://127.0.0.1:${handshake.port}`,
    token: handshake.token,
    pid: proc.pid!,
    proc,
    teardown() {
      proc.stdin?.end();
      proc.kill("SIGTERM");
    },
  };
}
```

**Using `VOSS_SERVE_FAKE_TURN`:** Setting this env var makes `_resolve_provider` return a fake resolution AND makes `_run_turn` emit a canned sequence (`user` → `thinking` → `plan` → `stream.delta` × 2 → `stream.finalize` → `final` → `session.idle`) without any provider calls. This is the correct CI seam — confirmed in `app.py:92` and `app.py:166`. [CITED: voss/harness/server/app.py — verified in repo]

**Fake turn emits exactly:** `server.connected` (SSE generator) → `user` → `thinking` → `plan` → `stream.delta("hello ")` → `stream.delta("from fake turn")` → `stream.finalize(role="assistant")` → `final(f"echo: {text}")` → `session.idle`. [CITED: voss/harness/server/app.py:167-177 — verified in repo]

### Anti-Patterns to Avoid

- **Hand-authoring event types.** The `type: Literal[...]` on each event class is the discriminator — it must stay in sync with the server. Codegen from the committed snapshot prevents drift. Never override generated types.
- **Using native `EventSource`.** It cannot send `Authorization: Bearer` headers and is absent in Node.js. Always use `fetch` body streaming + `eventsource-parser`.
- **Splitting the drift gate into REST-only.** The spec requires REST AND event-union to be covered. Both `openapi.json` and `events.schema.json` must be compared.
- **Making the exporter boot a live server.** Use `create_app(<fixed_token>).openapi()` — no `uvicorn`, no subprocess. This keeps the exporter and drift gate hermetic.
- **Importing `child_process` from `src/index.ts`.** The core entrypoint is browser-safe. Any spawn/process logic must live exclusively in `src/node.ts` and be imported only via the `"./node"` export.
- **Using `BaseHTTPMiddleware` for auth.** `app.py` uses raw ASGI (`_BearerASGI`) specifically because `BaseHTTPMiddleware` buffers response bodies and breaks SSE. The SDK does not implement its own server, but this explains why the SSE GET must carry `Authorization` — the middleware intercepts it before it reaches the route.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OpenAPI → TS types | Custom schema walker | `openapi-typescript` | Handles `oneOf`, `discriminator`, `$ref` resolution, `const` → literal, nullable, etc. |
| REST typed client | Custom fetch wrapper | `openapi-fetch` | Path/query/body type inference from generated `paths`; auth middleware; error model |
| SSE framing | Custom `\n\n`-split parser | `eventsource-parser` `EventSourceParserStream` | Handles partial chunks, multi-line data, comment lines, retry fields |
| Node child cleanup | `process.on('exit')` alone | `proc.stdin?.end()` + `SIGTERM` + `stdin` pipe | Server's `_watch_stdin_eof` fires on stdin close; SIGTERM handles the race; both needed |
| TypeScript build | Custom tsc config | `tsup` | Multi-entry, ESM output, `dts: true`, clean; one config file |
| Import graph enforcement | Manual code review | `dependency-cruiser` | Walks the full module graph including transitive; exits non-zero on violation |

**Key insight:** The hardest problem here is not the REST or SSE code — it's the byte-stable contract snapshot and the two CI drift gates. The exporter pattern is novel (no prior art in the repo for Python → JSON snapshot → TS codegen), so the plan must establish this infrastructure in Wave 0 before any client code is written.

## Common Pitfalls

### Pitfall 1: Non-Deterministic openapi.json Output

**What goes wrong:** Two consecutive runs of `create_app().openapi()` produce different JSON (different key order in `$defs`, different random token).

**Why it happens:** Default `json.dumps` does not sort keys. A randomly-generated token (via `secrets.token_urlsafe(32)`) appears in the schema if it leaks into the spec title or security scheme. Pydantic 2 `$defs` ordering can vary across Python versions.

**How to avoid:** Use `json.dumps(schema, indent=2, sort_keys=True)`. Use a FIXED token (`FIXED_TOKEN = "voss-contract-export-fixed-token-v1"`) in the exporter — not `None` which generates a random one. The token appears in `app.state.token` but not in the OpenAPI schema itself (no security scheme is registered); however, pinning it is defensive.

**Warning signs:** Drift gate fails on a freshly committed snapshot on re-run without any code change.

### Pitfall 2: `_force_event_schema` Not Called in Exporter Context

**What goes wrong:** The exported `openapi.json` is missing `EventEnvelope` and member schemas in `components/schemas`, so `openapi-typescript` generates no event union types.

**Why it happens:** `_force_event_schema` is called at the end of `create_app()`. If the exporter calls `app.openapi()` on a freshly-constructed app that did NOT go through `create_app`, the hook is missing.

**How to avoid:** Always call `create_app(<fixed_token>).openapi()` in the exporter — never construct `FastAPI()` directly. [CITED: voss/harness/server/app.py:262-450 — verified]

**Warning signs:** Generated `types.ts` contains only route types (`paths`) but no `EventEnvelope` / `StreamDelta` / `SessionIdle` etc.

### Pitfall 3: Pydantic 2 `oneOf` + discriminator Not Rendering as TS Literal

**What goes wrong:** openapi-typescript renders `AgentEvent` as `Record<string, unknown>` or as a generic object union without literal `type` fields, breaking `switch (event.type)` exhaustiveness.

**Why it happens:** If Pydantic 2 emits `discriminator` as a `propertyName` key without `const` on each member's `type` field, openapi-typescript may not emit literal types.

**How to avoid:** Verify the generated `contracts/openapi.json` manually: each member in the `oneOf` under `EventEnvelope` must have `"type": {"const": "<name>"}`. Pydantic 2 with `Literal["..."]` does emit `const`. If the schema is wrong, add `json_schema_extra={"const": "<type>"}` to each model. [ASSUMED — needs verification by checking the actual exported JSON]

**Warning signs:** TypeScript compiler reports `'type' cannot be used as a discriminant` or the switch has no exhaustion error on the `default` branch.

### Pitfall 4: `EventSourceParserStream` Availability

**What goes wrong:** `import { EventSourceParserStream } from 'eventsource-parser/stream'` fails with module not found.

**Why it happens:** The `./stream` subpath export is a separate entrypoint (confirmed: `exports["./stream"]` exists in eventsource-parser 3.1.0). Some bundlers need explicit `exports` map support.

**How to avoid:** Import from `eventsource-parser/stream` (not `eventsource-parser`). Ensure `tsup` targets ESM (already in D-07). Verify the package's exports map is respected. [VERIFIED: npm registry — exports map confirmed]

**Warning signs:** `ERR_PACKAGE_PATH_NOT_EXPORTED` or type error on import.

### Pitfall 5: Server Handshake Race

**What goes wrong:** The Node launcher connects before uvicorn's accept loop is ready, getting `ECONNREFUSED`.

**Why it happens:** Without the bind-before-serve pattern, there's a window between the handshake print and uvicorn accepting connections.

**How to avoid:** `serve.py` already calls `sock.listen(128)` BEFORE the handshake print. The OS queues connections during this window — uvicorn re-listens on the same socket harmlessly. The launcher just needs to read the handshake line and then connect; no sleep/retry needed. [CITED: voss/harness/server/serve.py:47-59 — verified]

**Warning signs:** `ECONNREFUSED` on first request immediately after handshake.

### Pitfall 6: `Authorization` Header on the SSE GET

**What goes wrong:** The SSE connection returns 401 silently — no visible error since SSE response bodies don't surface cleanly on 401.

**Why it happens:** `_BearerASGI` middleware intercepts ALL HTTP requests including the SSE GET before routing. The SSE GET at `GET /session/:id/events` must carry `Authorization: Bearer <token>`.

**How to avoid:** Always pass the token in the `headers` option of the `fetch` call that opens the SSE stream — not just for REST calls. [CITED: voss/harness/server/app.py:51-75 — verified]

**Warning signs:** Empty SSE stream, or immediate close without `server.connected`.

### Pitfall 7: `stdin` Must Be Piped for Server EOF Supervision

**What goes wrong:** The server does not self-terminate when the launcher exits, leaving an orphan `voss serve` process.

**Why it happens:** `serve.py` starts the `_watch_stdin_eof` daemon thread only if `not sys.stdin.isatty()`. If the launcher spawns `voss serve` without piping stdin (`stdio: ['inherit', ...]`), the stdin is a TTY and the daemon thread never starts.

**How to avoid:** Always spawn with `stdio: ["pipe", "pipe", "inherit"]` (pipe stdin, pipe stdout for handshake reading, inherit stderr for log visibility). The `stdin.end()` call on dispose triggers EOF. [CITED: voss/harness/server/serve.py:64-66 — verified]

**Warning signs:** `ps aux | grep voss serve` shows orphans after tests.

### Pitfall 8: TS Drift Gate — Timing of Codegen Step

**What goes wrong:** The TS drift gate (re-run `openapi-typescript`, diff against committed `types.ts`) runs before the Python drift gate, and the contracts are out of date.

**Why it happens:** Wave ordering issue — if the Python exporter hasn't run yet, `contracts/openapi.json` doesn't exist.

**How to avoid:** The plan must establish Wave 0 ordering: (1) Python exporter runs and commits `contracts/`, (2) codegen runs and commits `types.ts`, (3) both drift gates are written. [ASSUMED — plan must enforce this wave ordering explicitly]

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | Bearer token via `Authorization` header — generated per-server-process, compared with `secrets.compare_digest` on server |
| V3 Session Management | no | Token is ephemeral, loopback-only, process-scoped |
| V4 Access Control | no | Loopback-only; no multi-tenant surface |
| V5 Input Validation | yes (client) | Validate SSE event `type` against the discriminated union before use |
| V6 Cryptography | no | No TLS on loopback by design; token is `secrets.token_urlsafe(32)` |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Token leakage via process env | Information Disclosure | Token comes from handshake stdout, not env; do not log the token |
| Orphan `voss serve` processes | Denial of Service | Pipe stdin + SIGTERM on dispose; `getppid` poll on server side as fallback |
| Prototype pollution via `JSON.parse(event.data)` | Tampering | Validate parsed event against the generated union types; consider `Object.freeze` or Zod parse in high-security contexts [ASSUMED] |
| SSE event injection via crafted `data:` field | Tampering | Server controls the SSE stream (loopback + auth); threat only applies if server is compromised |
