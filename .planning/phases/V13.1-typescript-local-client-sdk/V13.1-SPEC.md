# Phase V13.1: TypeScript Local Client SDK — Specification

**Created:** 2026-06-06
**Ambiguity score:** 0.153 (gate: ≤ 0.20)
**Requirements:** 7 locked

## Goal

Ship an isomorphic TypeScript client SDK for the Voss local protocol: a **browser-safe core** (typed REST client + SSE typed-event client + permission-reply helpers + typed event union, all generated from the V13 contract snapshot) plus a **Node-only launcher/supervisor** that spawns and manages a `voss serve` child. The package builds and passes integration tests against a real `voss serve`; npm publish is out of scope.

## Background

Grounded in the current repo (scouted 2026-06-06):

- **The server + protocol exist and are LOCKED.** `voss serve` (`cli.py:3722`) → `serve.py:run_server` binds `127.0.0.1:0`, prints a one-line `{"v":1,"port":...,"token":...}` handshake to stdout, supervises via getppid-poll + stdin-EOF, and runs uvicorn. Endpoints (`server/app.py`): `POST/GET /session`, `GET /sessions/saved`, `GET/DELETE /session/:id`, `POST /session/:id/message`, `POST /session/:id/abort`, `GET /session/:id/cost`, `POST /session/:id/permission`, `GET /session/:id/events` (SSE), `GET /doctor`. Auth = `Authorization: Bearer <token>` on every request, 401 at middleware otherwise.
- **The event union is the SSE contract.** `server/events.py` defines the 21-member `AgentEvent` discriminated union (discriminator `type`) + `EventEnvelope`; first event `server.connected`, turn end `session.idle`; `permission.updated` blocks the gate.
- **PROTOCOL.md mirrors OpenCode's `packages/sdk/js`** where shapes are shared, so an OpenCode-style TS client degrades gracefully.
- **No TS SDK exists.** `npm/` is `@vosslang/cli` — the M6 bin-shim vendoring Python, NOT a library. `site/` is the Next.js marketing site (irrelevant). The TS SDK is a new sibling package.
- **No audit or session-tree endpoint exists yet** — audit is V9, session-tree is V4. The only session/cost surface today is `GET /session`, `GET /sessions/saved`, `GET /session/:id`, `GET /session/:id/cost`.
- **Hard upstream dependency:** V13 (foundation) produces the committed `openapi.json` + event-union schema this SDK's types are generated from, plus the drift gate. V13 must ship before V13.1 builds.

What does NOT exist yet: the `@vosslang/sdk` package, generated TS contract types, the REST/SSE clients, the permission helpers, and the Node launcher. Those are this phase's deliverables.

## Requirements

1. **Generated typed contract**: TS types are generated from the V13 contract snapshot, not hand-authored.
   - Current: No TS types exist; the contract lives only in Python (`events.py`, FastAPI routes) + (after V13) committed `openapi.json` + event schema.
   - Target: A codegen step emits TS types for all REST request/response bodies and the full `AgentEvent` tagged union, sourced from V13's committed `openapi.json` + event-union schema. The event union is discriminated on `type` (mirrors the Python/serde tag).
   - Acceptance: Regenerating types from the committed V13 snapshot produces the checked-in types byte-identically (deterministic); no event/route type in the SDK is hand-maintained; a TS consumer narrowing on `event.type` gets exhaustive type checking over all 21 events.

2. **Typed REST client**: Every protocol endpoint is callable with typed inputs/outputs.
   - Current: No client; callers would hand-roll `fetch` + Bearer auth.
   - Target: A REST client covering create/list/get/delete session, `sessions/saved`, post message, abort, cost, permission reply, and doctor — each typed from the generated contract, auto-attaching `Authorization: Bearer <token>` and the loopback base URL. REST errors surface as a typed error carrying HTTP status + `{v,detail}`.
   - Acceptance: Against a real `voss serve`, the client creates a session, posts a message (202), reads cost, and deletes the session (204); a 401 (bad token) and a 409 (message while a turn runs) each surface as the typed error with the correct status.

3. **SSE typed-event client**: The event stream is consumed as a typed async iterable of the tagged union.
   - Current: No SSE consumer.
   - Target: A client that opens `GET /session/:id/events`, yields typed `AgentEvent` values (each `data` parsed + validated against its `type`), and exposes the stream as an async iterator/event emitter. Handles the `server.connected` open, `session.idle` turn-end, and clean teardown on consumer abort (disconnect cancels the in-flight turn, per PROTOCOL §8).
   - Acceptance: Driving a `VOSS_SERVE_FAKE_TURN` (or real) turn, the client yields a `server.connected` first and a `session.idle` last, with `plan`/`stream.delta`/`final` typed in between; aborting the consumer mid-turn closes the stream without unhandled rejection.

4. **Permission-reply helpers**: A typed helper round-trips a permission gate.
   - Current: None.
   - Target: Given a `permission.updated` event (`{id, tool_name, args, dimension}`), a helper posts the reply to `POST /session/:id/permission` with a typed choice — `a` (allow once) / `A` (allow always) / `d` (deny), and `y`/`n` for scope-expand.
   - Acceptance: In an integration test, a turn that triggers a `fs_write` gate is unblocked by the helper replying `a`, and the subsequent `tool` event shows `state: ok`; a `d` reply blocks it.

5. **Node launcher / supervisor**: A Node entrypoint spawns and supervises a `voss serve` child.
   - Current: No launcher; the protocol's handshake/supervision is implemented server-side only.
   - Target: A Node-only API that spawns `voss serve` (stdin piped so the server's EOF supervision works), parses the one-line `{v,port,token}` handshake, exposes a configured client bound to that port+token, and kills the child on dispose/process-exit (kill-on-drop).
   - Acceptance: The launcher starts a server, returns a working client, runs a turn end-to-end, and on dispose leaves no orphan `voss serve` process (verified by PID check).

6. **Isomorphic packaging boundary**: The core is browser-safe; only the launcher is Node-bound.
   - Current: N/A (no package).
   - Target: `@vosslang/sdk` exposes a core entrypoint (REST + SSE + permission helpers + types) that imports zero `node:*` builtins, and a separate Node-only launcher entrypoint. The two are distinct exports so a bundler tree-shakes the launcher out of a browser build.
   - Acceptance: A lint/build check fails if any `node:*` or `child_process` import appears in the core entrypoint's module graph; the core entrypoint builds and type-checks for a browser target; the launcher entrypoint is reachable only via its own export path.

7. **Build + integration test harness**: The package builds and is tested against a real server.
   - Current: No package, no tests.
   - Target: `@vosslang/sdk` builds (TS → JS + d.ts) and ships a test suite that exercises REST + SSE + permission + launcher against a real `voss serve` (using the `VOSS_SERVE_FAKE_TURN` seam where no provider creds exist), runnable in CI.
   - Acceptance: `npm run build` produces dist + type declarations; the integration suite is green against a launched `voss serve` in CI; the suite covers all of requirements 2–5.

## Boundaries

**In scope:**
- New `@vosslang/sdk` package (sibling to `@vosslang/cli`)
- Generated TS contract types (from V13 snapshot)
- Typed REST client (all current endpoints)
- SSE typed-event client (async-iterable tagged union)
- Permission-reply helpers
- Node-only launcher/supervisor for `voss serve`
- Isomorphic split (browser-safe core, Node-only launcher entrypoint)
- Build + integration test harness against a real `voss serve`

**Out of scope:**
- npm publish / release pipeline — separate release decision (mirrors M6 gating publish behind its own pipeline)
- Audit & session-tree reader APIs — no such endpoint exists yet (audit=V9, session-tree=V4); readers limited to current `/session` + `/cost` surface, deep readers land when V9/V4 freeze
- A general in-process runtime SDK (embedding `voss_runtime` primitives in TS) — explicit non-goal; TS is a local *client*, not a runtime reimplementation
- Codegen *tool* choice (openapi-typescript / orval / etc.), fetch library, and SSE-parsing library — HOW, decided in discuss-phase
- VS Code extension / web UI built on the SDK — downstream consumers, not this phase
- Any change to `PROTOCOL.md` v1 or the server endpoints — V13.1 consumes the locked contract
- Browser-side launching of `voss serve` — impossible (no child processes in browser); web consumers attach to an already-running server

## Constraints

- **Hard dependency on V13** — types are generated from V13's committed `openapi.json` + event-union schema; V13.1 cannot build until V13 ships that snapshot. (V13.1 may be SPEC'd/planned ahead, but execution blocks on V13.)
- **Generated, not hand-written** — hand-authoring event/route types is forbidden; it reintroduces the drift V13's gate prevents.
- **Core imports zero `node:*`** — the REST/SSE/permission/types core must be browser-safe; `child_process` and other Node builtins live only behind the launcher entrypoint.
- **Integration tests run against a real `voss serve`** — the only faithful test of the handshake, Bearer auth, SSE framing, and supervision; use the `VOSS_SERVE_FAKE_TURN` seam to avoid provider credentials in CI.
- **Loopback + Bearer only** — no TLS (loopback), `Authorization: Bearer <token>` on every request including the SSE GET.
- Package name (`@vosslang/sdk`), module format (ESM/dual), codegen tool, and HTTP/SSE libraries are finalized in discuss-phase.
