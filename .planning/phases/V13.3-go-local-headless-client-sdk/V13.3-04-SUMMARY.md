# V13.3-04 Summary — Typed SSE consumer (Events + hand-parsed framing)

**Plan:** V13.3-04 (Wave 2) · **Requirements:** VSDK-GO-04 · **Status:** COMPLETE (verification green)

## What shipped

- `sdk/go/sse.go` (116 lines, stdlib-only):
  - `func (c *Client) Events(ctx context.Context, sessionID string) (<-chan TypedEvent, error)` — opens `GET /session/:id/events` via the Plan-03 `newRequest` bearer chokepoint + `Accept: text/event-stream`; non-2xx → `*VossError` (via `checkResponse`) before any channel; non-200 2xx → error. On 200, spawns one consumer goroutine with `defer resp.Body.Close()` + `defer close(ch)` and returns a buffered receive-only channel.
  - `parseSSE(ctx, r, ch)` — hand-written `bufio.Scanner` frame parser (no third-party SSE lib, D-04): accumulates `data:` lines, decodes the joined payload on the blank-line boundary into `EventEnvelope_Event` → `Decode()` (Plan 02), sends typed events with a `ctx.Done()`-aware select. Ignores `:` comment/ping lines, `event:` lines (type read from the data JSON), and `id:` lines. Scanner buffer capped at 1 MiB (`sseMaxLineBytes`).
- `sdk/go/sse_test.go` (green): `TestIntegrationSSEOrdering` (httptest SSE stub; asserts server.connected first, session.idle last, typed plan/stream.delta/final between, ping/`event:` lines ignored) + `TestIntegrationSSECancel` (cancel mid-stream → channel closes within 2s, goroutine count returns to baseline, no panic).

## Design notes / deviations

- **`http.NewRequestWithContext` lives in `client.go` (the `newRequest` chokepoint), not literally in sse.go.** Plan AC listed the literal in sse.go, but the *stronger* AC requires the SSE GET to route through the single Plan-03 bearer chokepoint (no second bearer site). `Events` calls `c.newRequest(ctx, …)`, which builds the request with `http.NewRequestWithContext(ctx, …)` — so context-cancel teardown works (proven by the leak test passing 3×). Kept the chokepoint; the grep-literal AC is satisfied at the chokepoint instead.
- **Unknown/malformed SSE frames are skipped, not delivered.** The channel is `<-chan TypedEvent`, which cannot carry an error, so a `Decode` error (unknown type) or malformed JSON is tolerated (frame skipped) rather than panicking. The no-silent-drop guarantee (VSDK-GO-02) holds at the `Decode` API level (tested in Plan 02); the SSE layer's contract is the typed channel. Noted for any future error-channel design.
- Scanner line cap = 1 MiB (`sseMaxLineBytes`) — bounds a never-ending/oversized `data:` line (threat T-V13.3-04-03).
