# Phase V13.3: Go Local / Headless Client SDK — Specification

**Created:** 2026-06-07
**Ambiguity score:** 0.133 (gate: ≤ 0.20)
**Requirements:** 8 locked

## Goal

Ship a reusable Go client module for the Voss local protocol — a **generated, full-fidelity event model + REST types** (sourced from the V13.1 committed contract snapshot), a **typed REST client**, a **typed SSE event consumer**, **handshake + bearer auth helpers**, **both a `voss serve` child supervisor (spawn) and an attach-to-existing constructor**, a **permission-reply helper**, and **protocol-exposed session/cost/doctor readers** — targeting headless/infra automation (CI bots, devops tooling, lightweight local runners). The module builds and passes integration tests against a real `voss serve`. Go module publishing is out of scope; no Python FFI and no reimplementation of orchestration semantics.

## Background

Grounded in the current repo (scouted 2026-06-07):

- **No Go SDK exists.** The only Go in the repo is a test fixture (`tests/fixtures/code/go/main.go`); there is no `go.mod`, no Go client, no codegen tooling (no `oapi-codegen` reference anywhere).
- **The server + protocol exist and are LOCKED.** `voss serve` (`voss/harness/cli.py:3722`) → `voss/harness/server/serve.py:run_server` binds `127.0.0.1:0`, prints a one-line `{"v":1,"port":...,"token":...}` handshake to stdout, supervises via getppid-poll + stdin-EOF heartbeat, runs uvicorn. `create_app(token)` (`server/app.py`) builds the FastAPI app. Endpoints: `POST/GET /session`, `GET /sessions/saved`, `GET/DELETE /session/:id`, `POST /session/:id/message`, `POST /session/:id/abort`, `GET /session/:id/cost`, `POST /session/:id/permission`, `GET /session/:id/events` (SSE), `GET /doctor`. Auth = `Authorization: Bearer <token>` on every request; 401 at middleware otherwise.
- **The event union is the SSE contract.** `voss/harness/server/events.py` defines the 21-member `AgentEvent` discriminated union (discriminator `type`) + `EventEnvelope` + `PROTOCOL_VERSION = 1`. `app.py:_force_event_schema` forces `EventEnvelope` into OpenAPI components "so codegen emits a tagged enum."
- **The contract snapshot is owned by V13.1, not V13.** V13 is **docs-only** (SDK Surface Matrix, stability tiers, language priority, `sdk.md`↔`PROTOCOL.md` reconciliation). The committed `openapi.json` snapshot + exporter + CI drift gate that all clients generate from are **V13.1 deliverables**; V13.2–.4 reuse that substrate. So V13.3's hard upstream dependency is **V13.1's committed snapshot + drift gate**.
- **Sibling precedent is locked.** V13.1 (TypeScript) and V13.2 (Rust) SPECs (both 2026-06-07) fixed the parallel boundary set: generated-not-hand-written types, full-fidelity canonical event union, publish deferred, deep audit/session-tree readers gated on V9/V4, real-server integration tests using the `VOSS_SERVE_FAKE_TURN` seam, no orchestration reimplementation, no PROTOCOL edits. V13.3 is the Go analog of that set.
- **Go-specific reality:** Go has no sum types, so the event union cannot be an exhaustive enum the way Rust's is. Representation (envelope + typed payloads decoded on demand vs. a typed struct + `type` switch) is an implementation choice for discuss-phase; the **fidelity bar** (all 21 members representable, no silent drop) is locked here.

What does NOT exist yet: the Go module, generated Go contract/event types, a typed REST client, a typed SSE consumer, handshake+bearer helpers, spawn + attach constructors, and a Go integration test suite against a launched `voss serve`. Those are this phase's deliverables.

## Requirements

1. **Generated typed contract** (VSDK-GO-01): Go protocol types are generated from the V13.1 contract snapshot, not hand-authored.
   - Current: No Go contract types exist; no codegen tooling in the repo. The contract lives in Python (`events.py`, FastAPI routes) + the committed `openapi.json` snapshot V13.1 produces.
   - Target: A codegen step emits Go types for the REST request/response bodies and a full-fidelity model of the 21-member `AgentEvent` union, sourced from V13.1's committed snapshot. Generated types are checked in and regenerate deterministically.
   - Acceptance: Regenerating types from the committed V13.1 snapshot reproduces the checked-in Go types byte-identically (deterministic); no REST/event type in the SDK is hand-maintained (enforced by a regeneration check in CI).

2. **Full-fidelity event model** (VSDK-GO-02): all 21 event members are representable; no silent loss.
   - Current: No Go event model. (`voss-tui`'s Rust `AppEvent` is lossy with an `Other` catch-all — explicitly not the bar here.)
   - Target: Every one of the 21 `AgentEvent` members is decodable into a typed Go value carrying its fields. An unknown/future `type` (e.g. a PROTOCOL v2 member) surfaces as an explicit error or sentinel value — never silently discarded. Go representation shape (envelope + `json.RawMessage` payload vs. flat typed struct + `type` switch) is decided in discuss-phase.
   - Acceptance: A decode test covers all 21 known `type` values into their typed payloads with fields populated; an unknown `type` produces a distinguishable error/sentinel (asserted), not a zero-value drop.

3. **Typed REST client** (VSDK-GO-03): every protocol endpoint is callable with typed inputs/outputs.
   - Current: No Go REST client.
   - Target: A client covering create/list/get/delete session, `sessions/saved`, post message, abort, cost, permission reply, and doctor — each typed from the generated contract, auto-attaching `Authorization: Bearer <token>` and the loopback base URL. Non-2xx responses surface as a typed Go error carrying the HTTP status + the server's `{v, detail}` body.
   - Acceptance: Against a real `voss serve`, the client creates a session, posts a message (202), reads cost, and deletes the session (204); a 401 (bad token) and a 409 (message while a turn runs) each surface as the typed error exposing the correct HTTP status.

4. **Typed SSE event consumer** (VSDK-GO-04): the event stream is consumed as typed events.
   - Current: No Go SSE consumer.
   - Target: A client that opens `GET /session/:id/events`, parses each SSE frame's `data`, decodes into the full-fidelity event model, and delivers events to the caller (idiomatic Go — channel or iterator). It observes the `server.connected` open and `session.idle` turn-end; cancelling the consumer (context cancel / Close) disconnects, which cancels the in-flight turn (disconnect-cancels, per PROTOCOL §8).
   - Acceptance: Driving a `VOSS_SERVE_FAKE_TURN` (or real) turn, the consumer delivers `server.connected` first and `session.idle` last, with `plan` / `stream.delta` / `final` typed in between; cancelling mid-turn ends delivery cleanly with no leaked goroutine (verified) and no panic.

5. **Spawn + attach client constructors** (VSDK-GO-05): the SDK both supervises a `voss serve` child and attaches to an existing one.
   - Current: No Go supervisor or attach path.
   - Target: (a) **Spawn:** a constructor that launches `voss serve` (stdin piped so the server's EOF-supervision heartbeat works), parses the handshake, returns a client bound to that port+token, and kills + reaps the child on `Close()`/context cancel (no orphan). Interpreter selection honors `VOSS_PYTHON` then repo `.venv/bin/python` then `python3`. (b) **Attach:** a constructor that takes a caller-supplied base URL + token (or a captured handshake line) and returns a client without owning any child process.
   - Acceptance: The spawn constructor starts a server, returns a working client, runs a turn end-to-end, and on `Close()`/cancel leaves no orphan `voss serve` process (PID/process-verified); a bad interpreter path surfaces as a typed spawn error rather than a hang. The attach constructor connects to a separately-launched server and completes a REST round-trip without spawning or killing any process.

6. **Handshake + bearer auth helpers** (VSDK-GO-06): the SDK owns the local-server handshake parse + bearer attach.
   - Current: None in Go.
   - Target: Helpers that (a) parse the one-line `{"v":1,"port":...,"token":...}` handshake into a typed value, and (b) produce a client pre-bound to the loopback base URL + bearer token, attaching `Authorization: Bearer <token>` to every request including the SSE GET. Server-handshake bearer only — distinct from LLM-provider credentials; no JSON-RPC bridge transport.
   - Acceptance: Given a captured handshake line, the helper yields the correct base URL + token; a request built by the SDK carries the `Authorization: Bearer <token>` header (integration test: correct token succeeds, wrong token → 401).

7. **Permission-reply helper + non-reimplementation guard** (VSDK-GO-07): permission gates are answerable; the SDK is a thin client.
   - Current: None in Go.
   - Target: A typed permission-reply helper (allow/deny choice → `POST /session/:id/permission`). The module contains only client/transport/codegen code — no port of EM/board/runtime orchestration semantics and no Python FFI/cgo.
   - Acceptance: The permission helper unblocks a gate with an allow reply and blocks with a deny reply (verified via the next `tool`/event state over SSE); `grep` of the Go module shows no `cgo`/FFI import and no reimplemented orchestration types (the module imports only the generated contract + an HTTP/SSE transport).

8. **Build + integration test harness** (VSDK-GO-08): the module builds and is tested against a real server.
   - Current: No `go.mod`, no Go tests.
   - Target: A Go module that builds with `go build ./...` and ships an integration test suite exercising REST + SSE + permission + spawn + attach against a real `voss serve` (using the `VOSS_SERVE_FAKE_TURN` seam where no provider creds exist), runnable in CI. Module path/location and codegen tool are decided in discuss-phase.
   - Acceptance: `go build ./...` and `go test ./...` are green; the integration suite covers requirements 3, 4, 5, and 6 against a launched `voss serve`; codegen determinism (req 1) and event-model fidelity (req 2) are covered by tests in the suite.

## Boundaries

**In scope:**
- New Go module (path/location decided in discuss-phase) targeting headless/infra automation
- Generated Go contract types + full-fidelity model of the 21-member `AgentEvent` union (from the V13.1 snapshot)
- Typed REST client (all current endpoints)
- Typed SSE event consumer (idiomatic Go delivery of the typed event model)
- Permission-reply helper
- Handshake + bearer auth helpers (local-server token, not provider creds)
- **Spawn** (`voss serve` child supervisor, kill-on-Close) **and attach** (connect to existing server) constructors
- Protocol-exposed readers (`sessions/saved`, `session`, `cost`, `doctor`)
- Build + integration test harness against a real `voss serve` (CI-runnable)

**Out of scope:**
- Go module publishing (module-proxy / `go install` discoverability) — separate release decision (mirrors V13.1/V13.2 deferred publish)
- Deep audit / session-tree / replay reader APIs — no such endpoint exists yet (audit = V9, session-tree = V4); readers limited to current `/session` + `/sessions/saved` + `/cost` + `/doctor`; deep readers land when V4/V9 freeze
- Reimplementing Python orchestration semantics (EM/board/runtime) in Go — explicit non-goal; Go is a local *client*
- Python FFI / cgo / embedding the interpreter — REST/SSE over loopback only
- The Go codegen tool choice, event-model representation shape, module/package layout, and SSE-delivery idiom (channel vs iterator) — HOW, decided in discuss-phase
- Any change to `PROTOCOL.md` v1 or the server endpoints — V13.3 consumes the locked contract
- Downstream devops daemons / CI bots built *on top of* the SDK — consumers, not this phase

## Constraints

- **Hard dependency on V13.1's committed snapshot + drift gate** — Go types are generated from the `openapi.json` + event-union schema V13.1 commits; V13.3 cannot generate its types until V13.1 ships that substrate. (V13.3 may be SPEC'd/planned ahead, but execution blocks on V13.1.)
- **Generated, not hand-written** — hand-authoring REST/event types is forbidden; it reintroduces the drift the V13.1 gate prevents.
- **Full fidelity, no silent loss** — all 21 members representable; an unknown future `type` is an explicit error/sentinel, not a dropped event.
- **Loopback + Bearer only** — no TLS (loopback), `Authorization: Bearer <token>` on every request including the SSE GET; server-handshake token only (not provider creds); no JSON-RPC bridge transport.
- **No Python FFI / cgo** — pure Go client over HTTP/SSE.
- **Integration tests run against a real `voss serve`** — the only faithful test of handshake, Bearer auth, SSE framing, and supervision; use the `VOSS_SERVE_FAKE_TURN` seam to avoid provider credentials in CI.
- **No orchestration reimplementation** — the module imports only the generated contract + an HTTP/SSE transport.
- Go module path/location, codegen tool (e.g. `oapi-codegen`), event-model representation, package layout, and SSE-delivery idiom are finalized in discuss-phase.
