## Deviations from plan (3)

**1. go floor `1.22` → `1.24` (FORCED).** Plan must-have/AC required `go 1.22`. Impossible: both locked, security-audited deps — oapi-codegen v2.7.1 AND oapi-codegen/runtime v1.4.1 — declare `go 1.24`, and Go raises the main module's go line to the max of dependency requirements. `runtime` is consumer-facing (imported by `types.gen.go`), so even a consumer needs 1.24. Keeping 1.22 would require downgrading to non-audited older dep versions — a worse deviation. The `^go 1.22` AC therefore fails by necessity; `go 1.24.3` is declared. The 1.22 floor was RESEARCH "Claude's discretion," not a CONTEXT-locked decision.

**2. Codegen runs through an in-SDK normalizer (NEW component, user-approved).** Plan assumed `//go:generate go tool oapi-codegen … ../../contracts/openapi.json` directly. oapi-codegen v2.7.1 cannot parse the **OpenAPI 3.1.0** FastAPI emits (issue #373) and fails on 3.1 null-union nullables (`anyOf: [{type:X},{type:null}]`, 9 sites). Added `internal/specgen` performing a deterministic in-memory 3.1→3.0 downgrade (version → `3.0.3`; single-type null unions → `nullable: true`) before delegating to the pinned `go tool oapi-codegen`. `oapi-codegen.yaml` stays the single config source; the `tool` directive + `tools.go` remain meaningful (specgen execs `go tool oapi-codegen`). Upstream `contracts/openapi.json` is never modified. User chose this approach over fixing V13.1's exporter or a python normalizer. `//go:generate` is now `go run ./internal/specgen` (still grep-matches `go:generate.*oapi-codegen`? No — matches `go:generate.*specgen`; the key-link `pattern: go:generate.*oapi-codegen` in the plan frontmatter is superseded by this approach).

**3. `types.gen.go` generated from `contracts/openapi.json` (real artifact), not the fixture.** Plan's D-08 premise ("snapshot does NOT exist on disk yet") is stale — `contracts/openapi.json` now exists (committed `8ea3e39`, requires `type` per-variant). Generating from the live contract keeps the now-active drift gate green. Verified the fixture and contracts produce **byte-identical** `types.gen.go` (key-ordering is the only input diff; oapi-codegen output is order-independent), so the fixture remains a valid documented fallback.
