## Summary

Phase V13.3 ships a greenfield Go client module at `sdk/go/` (module path `github.com/vosslang/voss/sdk/go`) with no existing Go infrastructure in the repo (only a test fixture at `tests/fixtures/code/go/main.go`; no `go.mod`). The server contract is fully locked and mapped here from live source files. The module provides generated types (from V13.1's `contracts/openapi.json`, not yet on disk), a hand-written REST client (stdlib `net/http`), a hand-parsed SSE consumer (stdlib `bufio.Reader`, channel delivery), a child-process supervisor that mirrors the proven Rust `server.rs` pattern, and both spawn and attach constructors.

The critical upstream dependency is V13.1 shipping `contracts/openapi.json` — execution blocks until that file exists on disk. Planning can proceed ahead; a concrete stub/fixture strategy is documented below for enabling pre-V13.1 test development.

The event model's 21-member union is the most nuanced Go-specific problem: oapi-codegen generates a `union json.RawMessage` wrapper type with `AsX()/FromX()` accessors, NOT a Go sum type or exhaustive enum. The recommended pattern (see Architecture Patterns §Event Model) wraps this with a hand-written `Decode(EventEnvelope) (TypedEvent, error)` dispatcher that (1) reads the discriminator, (2) calls the correct `AsX()` method, and (3) returns a typed error for any unknown `type` value — satisfying VSDK-GO-02's no-silent-drop requirement.

**Primary recommendation:** Use `oapi-codegen` v2.7.1 with `generate: models: true` (types-only — no generated client, no server stubs) over the V13.1 `contracts/openapi.json`. Hand-write the REST client and SSE consumer mirroring the Rust `net.rs` behavioral spec. Pin `oapi-codegen` via `go tool` directive (Go 1.24+ style). Wrap the generated `EventEnvelope` union with a `Decode()` dispatcher that returns typed events + error sentinel.

### REST Endpoints (from `voss/harness/server/app.py`)

| Method | Path | Status (success) | Status (error) | Request Body | Response Body |
|--------|------|-----------------|----------------|--------------|---------------|

| `GET` | `/session` | 200 | 401 | — | `{v, sessions:[{id,cwd,model,title,busy}]}` |
| `GET` | `/sessions/saved` | 200 | 401 | `?cwd=.` | `{v, sessions:[{id,name,cwd,model,updated_at,total_cost_usd,turns}]}` |

| `GET` | `/session/:id/cost` | 200 | 401, 404 | — | `{v, total_usd, turns}` |
| `POST` | `/session/:id/permission` | 200 | 401, 404 | `{id, choice}` | `{v, status:"ok"\|"stale"}` |
| `GET` | `/session/:id/events` | 200 (SSE) | 401, 404 | `Authorization: Bearer` header | `text/event-stream` |

**Auth middleware:** `_BearerASGI` (ASGI-layer, NOT `BaseHTTPMiddleware` — avoids SSE body buffering). Rejects any request without `Authorization: Bearer <token>` with `{v:1, detail:"unauthorized"}` + 401 BEFORE reaching route logic. Constant-time compare via `secrets.compare_digest`.

**409 trigger:** `POST /session/:id/message` while `session.busy == True` raises `HTTPException(409, "a turn is already running")`.

**Error body shape:** `{v:1, detail:"<message string>"}` (always JSON; matches PROTOCOL §9 error model).

**Note on `POST /session` status codes:** Returns 201 on success (not 200). Go client must check for `StatusCreated` (201) not `StatusOK` (200).

### SSE Event Stream Details (from `voss/harness/server/app.py` + `events.py`)

The SSE endpoint (`GET /session/:id/events`) uses `sse_starlette.EventSourceResponse` with:
- `ping=15` (keepalive ping every 15 seconds)
- `send_timeout=30`

**Frame format (sse-starlette / SSE spec):**
```
event: <type>
data: <json>
id: <seq>

```
(blank line between frames)

**First event:** `server.connected` is yielded immediately by the generator before entering the queue loop — no waiting for a turn to start.

**Disconnect-cancels:** When the SSE connection drops, `asyncio.CancelledError` propagates into `gen()`. The handler calls `s.task.cancel()` if a turn is running. This is the disconnect-cancels semantic (PROTOCOL §8) that the Go SDK's `context.Context` cancel must mirror.

**Queue source:** Events come from `s.queue` (an `asyncio.Queue` on `ServerSession`). The renderer (`EventBusRenderer`) puts events there thread-safely. The SSE `gen()` function is a pure async consumer of that queue.

### Handshake + Supervision (from `voss/harness/server/serve.py`)

`run_server()` (serve.py:39) does:
1. Creates an ephemeral TCP socket, binds `127.0.0.1:0`, calls `sock.listen(128)` — binds BEFORE printing handshake (race-free).
2. Generates a `token = secrets.token_urlsafe(32)` (URL-safe 32-byte base64; ~43 chars).
3. Prints exactly one JSON line to stdout: `{"v": 1, "port": <chosen_port>, "token": "<token>"}` (flushed).
4. If stdin is not a TTY (`not sys.stdin.isatty()`): starts a daemon thread (`_watch_stdin_eof`) that blocks reading stdin and sends `SIGTERM` on EOF.
5. Starts `asyncio` event loop; spawns `_watch_parent` task (polls `os.getppid()` every 2s; sends `SIGTERM` if PID changed, meaning parent died).

**Spawn command (from Rust `server.rs:56–58`):**
```
<python> -m voss.cli serve --port 0
```

**Stdin heartbeat:** The spawning process must hold the write-end of the stdin pipe open for the server's lifetime. Close the pipe (or let the pipe close when the Go process holding it exits) to trigger the EOF supervision. The Rust implementation uses `Stdio::piped()` and never writes to stdin; the pipe is held open by keeping the `Child.stdin` handle alive.

**SIGTERM handling:** `voss serve` runs under uvicorn. uvicorn handles SIGTERM gracefully (cancels in-flight turns). The Go SDK must send SIGTERM (not SIGKILL) first, then wait, then SIGKILL if timeout. However, the Rust implementation uses `child.start_kill()` (SIGKILL directly via tokio) followed by `child.wait()`. For the Go SDK, `cmd.Process.Kill()` sends SIGKILL on Unix. Either is acceptable given the server self-terminates on SIGTERM (getppid poll) and the test harness uses FAKE_TURN which completes immediately.

### `VOSS_SERVE_FAKE_TURN` (from `voss/harness/server/app.py:38–50, 166–178`)

**Location:** `app.py:_resolve_provider()` (line 92–93) checks `os.environ.get("VOSS_SERVE_FAKE_TURN")` and returns `_FakeResolution()` + a placeholder `object()`. `_run_turn()` (line 166–178) checks the same env var and emits a canned event sequence:

```
user(task="<text>")
thinking(label="planning 1/1")
plan(confidence=0.9, steps=[], cost_usd=0.0)
stream.delta(text="hello ")
stream.delta(text="from fake turn")
stream.finalize(role="assistant", confidence=0.9, cost_usd=0.0)
final(text="echo: <text>", confidence=0.9, cost_usd=0.0)
session.idle(session_id=<id>)
```

All emitted over the real SSE path (real `EventBusRenderer`, real queue, real `EventSourceResponse`). The `server.connected` event is emitted by the SSE endpoint generator itself (before the queue loop), not by the fake turn.

**Full sequence seen by a Go SSE consumer:**
1. `server.connected`
2. `user`
3. `thinking`
4. `plan`
5. `stream.delta` (×2)
6. `stream.finalize`
7. `final`
8. `session.idle`

**Implication for tests:** `VOSS_SERVE_FAKE_TURN=1` must be set in the spawned process's environment. The Go `spawn_server_with` analog accepts extra env vars (same as Rust `spawn_server_with(python, extra_env)`). No provider creds needed — `POST /session` succeeds (fake provider satisfies the route), `POST /session/:id/message` (202) triggers the fake turn, SSE stream delivers 8 events total.

**Note:** `session.busy` is still set/cleared by `_run_turn` even in fake mode (`session.task = None` in `finally`). So the 409 test requires a real concurrent turn or a timing fixture; the fake turn is synchronous from the server's asyncio perspective (runs to completion immediately). Testing 409 may require posting a second message before the fake turn's `session.idle` is delivered — or using a sleep/delay in the fake turn body (which requires a server patch not in scope). **Alternative:** test 409 by having two concurrent Go goroutines post messages; the race is reliable since the server is single-threaded per session. Document this in tests.

### Permission Protocol (PROTOCOL §7 + app.py:379–386)

**Wire shape of `permission.updated` event:**
```json
{"v":1, "type":"permission.updated", "id":"<8-hex>", "tool_name":"<name>", "args":{...}, "dimension":"tool|confidence|budget"}
```

**Permission reply endpoint:** `POST /session/:id/permission`
- Request: `{id: str, choice: str}` where `choice ∈ "a" | "A" | "d" | "y" | "n"`
- Response: `{v:1, status:"ok" | "stale"}`
- HTTP 200 always (unless 401/404)

**"stale" response:** If the Future is already resolved (timeout or double-reply), status is "stale". Not an error from the HTTP perspective — the Go SDK should expose this as a successful reply with a `Stale` boolean.

**Timeout:** 300 seconds (PERMISSION_TIMEOUT_S). Server auto-denies ("d") on timeout.

**`dimension` values:** `"tool"` (normal tool approval), `"confidence"` (confidence gate), `"budget"` (budget gate).

### Pattern 2: Event Model — Decode Dispatcher (Claude's Discretion recommendation)

**What:** oapi-codegen generates `EventEnvelope` with a `union json.RawMessage` field (not typed) plus `AsServerConnected()`, `AsSessionIdle()`, ..., `AsGateUpdated()` accessor methods (one per union member). There is also a `Discriminator()` method that unmarshals the raw bytes to read the `type` field.

**The gap:** oapi-codegen does NOT generate an exhaustive `switch` statement or a `Decode()` function. The caller must invoke `Discriminator()` then call the correct `AsX()` method. An unknown `type` value means none of the `AsX()` methods match — this is where silent drop can occur.

**Recommended pattern (hand-written `Decode()` in `events.go`):**

```go
// Source: design derived from oapi-codegen union generation + VSDK-GO-02 requirement

// TypedEvent is the sealed interface all 21 typed event structs implement.
type TypedEvent interface{ eventType() string }

// ErrUnknownEventType is returned by Decode for unrecognized `type` values.
// It carries the raw type string so callers can log/forward without silent drop.
type ErrUnknownEventType struct{ Type string }
func (e ErrUnknownEventType) Error() string { return "unknown event type: " + e.Type }

// Decode dispatches an EventEnvelope to its typed Go value.
// Returns ErrUnknownEventType for any `type` not in the 21-member set —
// never returns nil error with a nil event (satisfies VSDK-GO-02).
func Decode(env EventEnvelope) (TypedEvent, error) {
    disc, err := env.event.Discriminator()  // reads "type" field from raw JSON
    if err != nil {
        return nil, fmt.Errorf("event discriminator: %w", err)
    }
    switch disc {
    case "server.connected":
        v, err := env.event.AsServerConnected(); return v, err
    case "session.idle":
        v, err := env.event.AsSessionIdle(); return v, err
    case "permission.updated":
        v, err := env.event.AsPermissionUpdated(); return v, err
    // ... all 21 cases ...
    case "gate.updated":
        v, err := env.event.AsGateUpdated(); return v, err
    default:
        return nil, ErrUnknownEventType{Type: disc}
    }
}
```

**Why this satisfies VSDK-GO-02:** Any unknown `type` (e.g., a PROTOCOL v2 member) hits `default` and returns a typed, non-nil error. The caller can distinguish `ErrUnknownEventType` from other errors via `errors.As`. No event is silently dropped.

**Why not use `Discriminator()` only:** Because that gives a string, not typed data. The typed Go values (with fields) are needed for VSDK-GO-02's "decodable into typed Go value carrying its fields" requirement.

### Pattern 3: stdlib SSE Parsing (D-04 compliant)

**What:** Parse SSE frames from `http.Response.Body` using `bufio.Reader`. Honor `context.Context` cancel to abort the in-flight SSE GET (which cancels the server-side turn per PROTOCOL §8).

**Key correctness points:**
1. The `http.Request` must be created with `http.NewRequestWithContext(ctx, ...)` — NOT `http.NewRequest`. This ensures `ctx.Done()` cancels the underlying TCP read.
2. On `ctx.Done()`, the `bufio.Scanner` / `bufio.Reader` read returns an error. The SSE consumer goroutine exits, which allows garbage collection. No manual `resp.Body.Close()` race is needed — the context cancellation tears down the transport.
3. The SSE frame parser must handle: `event: <type>`, `data: <json>`, blank-line frame separator. The `id:` line can be ignored for consumption purposes.
4. Channel delivery: the goroutine writes to `chan TypedEvent` and exits when the context is cancelled or the channel is closed. The caller reads from the channel.

**Frame format (per PROTOCOL §6 and sse-starlette):**
```
event: server.connected\r\n
data: {"v":1}\r\n
id: 1\r\n
\r\n
```
Lines may be terminated with `\r\n` or `\n` (SSE spec allows both). The parser must strip both.

**Minimal correct Go pattern:**
```go
// Source: Go stdlib bufio, net/http, context documentation [ASSUMED pattern]

func (c *Client) Events(ctx context.Context, sessionID string) (<-chan TypedEvent, error) {
    req, err := http.NewRequestWithContext(ctx, "GET",
        c.baseURL+"/session/"+sessionID+"/events", nil)
    if err != nil { return nil, err }
    req.Header.Set("Authorization", "Bearer "+c.token)
    req.Header.Set("Accept", "text/event-stream")

    resp, err := c.http.Do(req)
    if err != nil { return nil, err }
    if resp.StatusCode != http.StatusOK {
        resp.Body.Close()
        return nil, parseErrorBody(resp)
    }

    ch := make(chan TypedEvent, 32) // buffered to avoid blocking the parser goroutine
    go func() {
        defer resp.Body.Close()
        defer close(ch)
        parsSSEStream(ctx, resp.Body, ch)
    }()
    return ch, nil
}
```

**No goroutine leak verification (SPEC AC):** Use `goleak` or check that the channel is closed after the context is cancelled. The pattern above guarantees channel close via `defer close(ch)` + goroutine exit via `resp.Body.Close()` / context cancellation of the underlying TCP read.

[CITED: pkg.go.dev/net/http, pkg.go.dev/bufio, pkg.go.dev/context]

### Pattern 4: Child Supervisor (no-orphan)

**What:** Spawn `voss serve` using `os/exec.Cmd`, hold stdin pipe open as a heartbeat, kill + reap on `Close()`.

**Rust behavioral spec to mirror (from `server.rs`):**
1. Build command: `<python> -m voss.cli serve --port 0`
2. Set `stdin = Stdio::piped()` (held open, never written to)
3. Set `stdout = Stdio::piped()` (read one JSON line for handshake)
4. Set `stderr = Stdio::inherit()` (show server errors)
5. Set `kill_on_drop = true` (Rust-specific; Go equivalent: explicit `Kill()+Wait()` in `Close()`)
6. Spawn; read stdout lines until one parses as `{"port":N,"token":"T"}` within 20s timeout
7. Drain remaining stdout in a background goroutine
8. Return `ServerHandle{child, base:"http://127.0.0.1:N", token:"T"}`
9. On `shutdown()`: `child.start_kill(); child.wait().await`

**Go equivalent pattern:**
```go
// Source: behavioral mirror of crates/voss-tui/src/server.rs [ASSUMED Go idiom]

type SpawnHandle struct {
    cmd     *exec.Cmd
    stdinW  io.WriteCloser  // held open = heartbeat; Close() triggers EOF supervision
    BaseURL string
    Token   string
}

func Spawn(ctx context.Context) (*Client, error) {
    python := interpreterPath()
    cmd := exec.CommandContext(ctx, python, "-m", "voss.cli", "serve", "--port", "0")
    
    stdinR, stdinW, _ := os.Pipe()
    cmd.Stdin  = stdinR         // server reads from this (EOF kills server)
    cmd.Stdout = &bytes.Buffer{} // captured below for line reading
    cmd.Stderr = os.Stderr

    stdoutPipe, _ := cmd.StdoutPipe()
    cmd.Stdout = nil // use StdoutPipe instead
    
    if err := cmd.Start(); err != nil { return nil, &SpawnError{Err: err} }
    stdinR.Close() // parent side of the reader; server has the read end via cmd.Stdin

    // Actually: use StdoutPipe() before Start(), then cmd.Start(), then read.
    hs, err := readHandshake(stdoutPipe, 20*time.Second)
    // ...
}

func interpreterPath() string {
    if v := os.Getenv("VOSS_PYTHON"); v != "" { return v }
    venv := filepath.Join(repoRoot(), ".venv", "bin", "python")
    if _, err := os.Stat(venv); err == nil { return venv }
    return "python3"
}
```

**No-orphan guarantee:** Call `cmd.Process.Kill()` then `cmd.Wait()` in `Close()`. `cmd.Wait()` reaps the zombie. The stdin pipe being held open means the server's `_watch_stdin_eof` thread blocks; closing `stdinW` (the write end) causes the server to receive EOF on stdin and send SIGTERM to itself, which gracefully shuts down uvicorn. For immediate teardown, `Kill()` is acceptable (avoids waiting for graceful shutdown).

**PID verification for test:** After `Close()`, check that `/proc/<pid>/status` (Linux) or `kill(pid, 0)` returns `ESRCH` (no such process). On macOS: `os.FindProcess(pid).Signal(os.Signal(0))` returns an error when the process is gone.

**repo root discovery for `.venv`:** The Rust implementation uses `env!("CARGO_MANIFEST_DIR")` + `"../../.venv/bin/python"` (two levels up from `crates/voss-tui` to repo root). The Go equivalent: at runtime, find `go.mod` or use a compile-time embed of the repo root path. Simplest: use the module path `sdk/go/` — two levels up is the repo root. OR just try `../../.venv/bin/python` relative to the binary's working directory, and fall back to `python3` if it doesn't exist (same logic Rust uses).

### Anti-Patterns to Avoid

- **Silent event drop:** Reading a `type` string from the SSE frame without calling `Decode()` and just ignoring unknown types. Must return `ErrUnknownEventType`.
- **Closing stdin immediately after spawn:** The server uses EOF supervision — closing `stdinW` before `Close()` is called will cause the server to self-terminate mid-session.
- **Using `resp.Body.Close()` in the SSE goroutine without context:** If the goroutine just ranges over SSE events without a context, it will leak until the server closes the connection. Always use `http.NewRequestWithContext`.
- **Forgetting `cmd.Wait()` after `cmd.Process.Kill()`:** Without `Wait()`, the child becomes a zombie on Linux/macOS.
- **Hand-authoring REST/event types:** SPEC constraint; regeneration check will fail.
- **Importing `cgo` or `unsafe`:** VSDK-GO-07 grep check will fail CI.

## Common Pitfalls

### Pitfall 1: `POST /session` returns 201, not 200

**What goes wrong:** Go `net/http` client checks `resp.StatusCode == http.StatusOK` (200); the create-session endpoint returns 201 (`StatusCreated`).
**Why it happens:** FastAPI `@app.post("/session", status_code=201)` — this is correct per REST conventions for resource creation.
**How to avoid:** Check `resp.StatusCode == http.StatusCreated` for `POST /session`; check `resp.StatusCode == http.StatusNoContent` for `DELETE /session/:id`; check `resp.StatusCode == http.StatusAccepted` for `POST /session/:id/message` and `POST /session/:id/abort`.
**Warning signs:** "unexpected status 201" errors in early test runs.

### Pitfall 2: SSE goroutine leak on context cancel

**What goes wrong:** The SSE goroutine blocks on `bufio.Scanner.Scan()` and never exits when the context is cancelled, because the HTTP response body doesn't close until explicitly closed.
**Why it happens:** `bufio.Scanner` / `bufio.Reader` blocks on I/O. Context cancel does NOT automatically close the response body unless the request was made with `http.NewRequestWithContext` AND the transport cancels the underlying TCP connection.
**How to avoid:** Always use `http.NewRequestWithContext(ctx, ...)`. Verify with `goleak` in tests.
**Warning signs:** Goroutine count grows with each cancelled SSE consumer.

### Pitfall 3: Handshake line may follow non-JSON server startup lines

**What goes wrong:** Parser looks only at the first stdout line from `voss serve`, which may be a uvicorn startup log line, not the JSON handshake.
**Why it happens:** `uvicorn` sometimes prints `INFO: Started server process [PID]` before the server code's `print(json.dumps({...}))` — except that `serve.py` prints the handshake BEFORE calling `uvicorn.Server.serve()`, and sets `log_level="warning"`. So INFO logs are suppressed. However, future changes or different environments may add earlier output.
**How to avoid:** Mirror the Rust approach: scan each line, attempt `json.Unmarshal` into `Handshake{Port, Token}`, accept the first line that parses. Apply a 20-second timeout (same as Rust).
**Warning signs:** "server handshake timed out" when uvicorn starts slowly.

### Pitfall 4: `PrinciplesOverflow` is not in PROTOCOL.md but IS in events.py

**What goes wrong:** PROTOCOL.md §6 lists 20 event types (no `principles_overflow`). If codegen is tested against the PROTOCOL table only, this member may be omitted from test coverage.
**Why it happens:** PROTOCOL.md is slightly stale relative to `events.py` source.
**How to avoid:** Test all 21 members using `events.py` as the authoritative list (which is what `_force_event_schema` exposes via the live `openapi.json`). The planner should include a decode-fidelity test for `principles_overflow`.
**Warning signs:** Test coverage says "20/20" when it should say "21/21."

### Pitfall 5: stdin pipe — must hold write end open

**What goes wrong:** `cmd.Stdin = stdinPipe` is set, but the write end of the pipe is closed immediately after `cmd.Start()`. This causes the server to immediately detect stdin EOF and SIGTERM itself.
**Why it happens:** Pipe has two ends. The parent must keep the WRITE end open. Common mistake: calling `stdinW.Close()` right after `Start()` thinking "we don't need to write anything."
**How to avoid:** Store `stdinW` in `SpawnHandle`. Only close it in `Close()` / defer.
**Warning signs:** Server exits immediately after spawn; handshake timeout.

### Pitfall 6: `oapi-codegen` union representation for `type`-discriminated events

**What goes wrong:** Assuming oapi-codegen generates a Go sum type or exhaustive enum for the 21-member union. It generates a `union json.RawMessage` struct with `AsX()` methods — NOT a `switch`-safe Go enum.
**Why it happens:** Go has no algebraic data types. oapi-codegen cannot generate them.
**How to avoid:** The `Decode()` function in `events.go` is a HAND-WRITTEN wrapper (NOT generated) that adds the exhaustive switch + unknown-type error. This file is committed alongside `types.gen.go` and is NOT regenerated. Make this distinction clear in the plan.
**Warning signs:** Trying to range-switch over the generated type without calling `Discriminator()` first.

### Pitfall 7: 409 test requires a running turn

**What goes wrong:** `VOSS_SERVE_FAKE_TURN` completes instantly (no `await asyncio.sleep`). Testing 409 by posting two messages sequentially in the same goroutine always gets the second message AFTER the first turn completes (session.idle received), so no 409 is triggered.
**Why it happens:** The fake turn runs to completion synchronously within the asyncio event loop before the second HTTP request is processed.
**How to avoid:** Test 409 by posting a message with a real provider and a second message before SSE session.idle — OR by using a patched fake turn that includes a sleep. The simplest approach for the integration test: check that when `session.busy = True` in the live server, a second message returns 409. Use `VOSS_SERVE_FAKE_TURN` + a short `time.Sleep` between the two posts (risky for flakiness) OR test 409 by examining the server's response when `busy=True` via a custom helper that checks the state directly.

**Recommended approach:** Post message, verify 202, immediately post again without waiting for SSE session.idle, check that the second post returns 409. The fake turn is fast but not instantaneous from the HTTP transport perspective (two HTTP round-trips happen faster than one async turn completes IF done from separate concurrent goroutines). Use `sync.WaitGroup` to fire both posts simultaneously.

## Code Examples

### Handshake Parsing

```go
// Source: behavioral mirror of crates/voss-tui/src/server.rs Handshake struct [ASSUMED]

type Handshake struct {
    V     int    `json:"v"`
    Port  uint16 `json:"port"`
    Token string `json:"token"`
}

func readHandshake(r io.Reader, timeout time.Duration) (Handshake, error) {
    ch := make(chan Handshake, 1)
    errCh := make(chan error, 1)
    go func() {
        scanner := bufio.NewScanner(r)
        for scanner.Scan() {
            var h Handshake
            if err := json.Unmarshal(scanner.Bytes(), &h); err == nil && h.Token != "" {
                ch <- h
                return
            }
        }
        if err := scanner.Err(); err != nil {
            errCh <- err
        } else {
            errCh <- errors.New("server exited before handshake")
        }
    }()
    select {
    case h := <-ch: return h, nil
    case err := <-errCh: return Handshake{}, err
    case <-time.After(timeout): return Handshake{}, errors.New("handshake timeout")
    }
}
```

**If A1 is wrong:** The `Decode()` function reads the discriminator by unmarshaling a minimal `struct{ Type string \`json:"type"\` }` directly from the raw bytes — same result, different approach. Unblocking.

## Open Questions (RESOLVED)

> All three carried into the plan. **Q1** RESOLVED via Plan V13.3-01 Wave 0 `discriminator_probe_test.go` (empirically verifies the generated `Discriminator()` signature before Plan 02's `Decode()` is finalized). **Q2** RESOLVED via Plan V13.3-06's concurrent-goroutine `TestIntegration409` (documented timing approach + flaky-fallback note). **Q3** RESOLVED via a unit decode test covering all 21 members incl. `principles_overflow` (Plan V13.3-02), independent of the FAKE_TURN SSE path which does not emit it.

   - What's unclear: Exact method signature of `Discriminator()` — does it return `(string, error)` or just `string`?
   - Recommendation: Wave 0 includes a task to generate types from the fixture and verify the exact `Discriminator()` signature; adjust `Decode()` accordingly. This is a 5-minute verification after codegen runs.

2. **409 test timing**
   - What we know: `VOSS_SERVE_FAKE_TURN` completes quickly; 409 requires `session.busy == True` at second-POST time.
   - What's unclear: Can two concurrent Go HTTP posts reliably land before the fake turn's `session.idle` clears `session.task`?
   - Recommendation: Use concurrent goroutines (`sync.WaitGroup`) to fire two messages simultaneously; the race is reliable in practice. Document as a timing-sensitive test.

3. **`PrinciplesOverflow` in FAKE_TURN path**
   - What we know: The `FAKE_TURN` sequence does not emit `principles_overflow`. Testing this event type requires either a real turn or a server-side modification.
   - Recommendation: Test `principles_overflow` decode in the unit decode test (construct fixture JSON, call `Decode()`); do not require it in the integration SSE test.

## Security Domain

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes (client-side) | Bearer token from handshake; attached to every request; no storage |
| V3 Session Management | no | Server owns session state; client is stateless |
| V4 Access Control | no | Server enforces; client sends the token |
| V5 Input Validation | partial | Handshake JSON parse validates required fields; REST request bodies typed |
| V6 Cryptography | no | Loopback only; no TLS; token is server-generated |

### Threat Patterns for This Stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Token in environment variable | Information Disclosure | `VOSS_PYTHON` is process-level; token is ephemeral per-server-process; not persisted |
| Orphan child process holding port | Denial of Service | `cmd.Process.Kill()` + `cmd.Wait()` on all exit paths (defer in `SpawnClient.Close()`) |
| Stdin pipe not closed | Denial of Service (server zombie) | `stdinW.Close()` in `defer` ensures server's EOF supervision fires |
| SSE goroutine leak | Resource exhaustion | `http.NewRequestWithContext` + defer `resp.Body.Close()`; goleak test |
