<decisions>
## Implementation Decisions

### Codegen toolchain
- **D-01:** Types generated with **`typify`** (Oxide; JSON Schema → serde Rust types). One pass over the event-union schema (`EventEnvelope.model_json_schema()`) yields the `#[serde(tag = "type")]` `AgentEvent` enum (all 21 members); a pass over the OpenAPI components yields the REST request/response structs. `progenitor` (full reqwest-client generator) REJECTED: its generated client shape doesn't match the loopback+bearer+supervisor+SSE needs, SSE isn't OpenAPI-modeled so the stream is hand-written anyway, and it adds a heavier build surface for little gain. Hand-rolling serde REJECTED (SPEC "generated, not hand-written" constraint).
- **D-02:** The thin REST + SSE client is **hand-written**, modeled directly on `voss-tui`'s proven `net.rs` (`HttpClient` methods + `stream_events`). Codegen provides the *types*; the client wiring (bearer auth, base URL, `{detail}` error mapping, SSE framing) is hand-authored.

### HTTP + SSE + async stack
- **D-03:** Reuse `voss-tui`'s proven stack via **workspace dependency pins**: `reqwest` (rustls-tls, `stream` feature), `eventsource-stream` 0.2, `futures-util`, `tokio` (full). No new/divergent HTTP or SSE crates. `reqwest-eventsource` stays REJECTED (pins an incompatible reqwest — same reason voss-tui avoided it).
- **D-04:** SSE event client is exposed as an async `Stream<Item = AgentEvent>` (`impl Stream`), not the mpsc-channel shape `voss-tui` uses internally — a library-grade surface. Connect-timeout bounds establishment without capping the long-lived stream body (mirror `net.rs` `HttpClient::new`). Dropping the stream cancels the in-flight turn (PROTOCOL §8).

### Crate layout
- **D-05:** Crate name `voss-sdk`, located at **`crates/voss-sdk`** (conventional Rust-workspace placement alongside `voss-tui`/`voss-auth`/`voss-bridge`), added to root `Cargo.toml` `members`. Easy path-dep for the future `voss-tui` dedupe. (Diverges from V13.1's `sdk/typescript/` grouping — Cargo-workspace convention wins for Rust.)
- **D-06:** Generated Rust types are **checked in** (committed), not build-time-only — keeps the crate buildable/inspectable without re-running codegen and enables the byte-identical drift check. (Mirrors V13.1 D-11.)

### Snapshot dependency + drift gate
- **D-07:** **Hard execution dependency on V13.1**, NOT V13. V13.1 OWNS the committed contract snapshot — `contracts/openapi.json` (`create_app(<fixed-token>).openapi()`) + `contracts/events.schema.json` (`EventEnvelope.model_json_schema()`) + the deterministic Python exporter + the Python drift gate (V13.1 D-10). V13 is docs-only (strategy/matrix/tiers/sdk.md/Python-M7) and is NOT a build dependency. ⚠ **Corrects the V13.2-SPEC.md "depends on V13" wording** — the real source of the generated types is V13.1's `contracts/` snapshot. SPEC requirements 1 & the V13-dependency constraint should be read with this correction; the snapshot artifacts do not exist on disk yet (V13.1 unexecuted as of 2026-06-07).
- **D-08:** Cargo-side drift gate: a test (or `cargo xtask`-style step) regenerates the Rust types from the committed `contracts/` snapshot and fails if `git diff` on the generated file is non-empty (mirrors V13.1 D-13b; SPEC AC "regenerate byte-identically"). The Python-side any-diff snapshot gate stays owned by V13.1.
- **D-09:** V13.2 does **not** vendor its own exporter (rejected the self-export option) — single source of truth is V13.1's `contracts/`. If V13.1 hasn't shipped the snapshot when V13.2 planning starts, execution blocks on it; planning may proceed ahead.

### Claude's Discretion
- Public API ergonomics (method names, builder vs free functions, error enum shape, module boundaries), the exact `typify` invocation (build.rs vs committed-codegen step vs `cargo xtask`), test-fixture wiring for `VOSS_SERVE_FAKE_TURN`, and where the lossy `AppEvent`-parity projection type lives — left to research/planning within the SPEC's acceptance criteria.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `crates/voss-tui/src/net.rs::HttpClient` — full REST surface already implemented: `create_session`, `create_session_resume`, `post_message`, `abort`, `permission_reply`, `list_saved_sessions`, `cost`, `doctor`, `stream_events`. Promote to a library-grade typed client.
- `crates/voss-tui/src/server.rs::{ServerHandle, spawn_server, spawn_server_with, python_path}` — the supervisor, including `VOSS_PYTHON` → `.venv/bin/python` → `python3` interpreter resolution and stdin-heartbeat/kill-on-drop. Lift nearly as-is.
- `VOSS_SERVE_FAKE_TURN` env seam (server `_run_turn`) — emits a canned turn over the real event/SSE path with no provider creds. The integration-test driver for CI (same seam V13.1 uses).
- V13.1's `contracts/{openapi.json,events.schema.json}` (once shipped) — the typify input.

### Established Patterns
- Handshake-then-bind (`serve.py` + `server.rs`): spawn `voss serve --port 0` with stdin PIPED (EOF supervision), read one stdout JSON line `{v,port,token}`, then `http://127.0.0.1:<port>` + `Authorization: Bearer <token>` on every request incl. the SSE GET.
- SSE framing per PROTOCOL §6: `event: <type>` / `data: {...}`; first event `server.connected`, turn end `session.idle`; client disconnect cancels the in-flight turn (§8). `voss-tui` decodes via `eventsource-stream` `.bytes_stream().eventsource()`.
- Non-2xx → error carrying server `{detail}` (`net.rs::ok_or_detail`); the SDK's typed error mirrors this + the HTTP status.
- Permission round-trip (PROTOCOL §7): `permission.updated {id,tool_name,args,dimension}` → reply `POST /session/:id/permission` choice ∈ `a|A|d` (+ `y|n` scope).

### Integration Points
- REST endpoints (app.py): `POST/GET /session`, `GET /sessions/saved`, `GET/DELETE /session/:id`, `POST /session/:id/message`, `POST /session/:id/abort`, `GET /session/:id/cost`, `POST /session/:id/permission`, `GET /session/:id/events` (SSE), `GET /doctor`.
- Root `Cargo.toml` `members` — add `crates/voss-sdk`; reuse `[workspace.dependencies]` pins.

</code_context>

<specifics>
## Specific Ideas

- The canonical SDK event type has **no `Other` catch-all** — it is the exhaustive 21-member generated union (a non-exhaustive `match` is a compile error). `voss-tui`'s lossy `AppEvent` (with `Other`) is reproduced only by the SDK's separate parity-projection (SPEC req 4), keeping the two representations distinct while voss-tui stays byte-stable.
- The auth concern split is deliberate: SDK owns the **local-server handshake bearer token** only; `voss-auth` (provider creds) and `voss-bridge` (JSON-RPC) are explicitly NOT reused — different channels.

</specifics>
