## Summary

`voss-sdk` is a new library crate (`crates/voss-sdk`) that lifts the proven client surfaces from the `voss-tui` binary crate into a reusable public API. The core technical challenge is **codegen**: the event-union schema Pydantic v2 emits carries `"type": { "const": "..." }` on each variant but does NOT include `"type"` in the per-variant `"required"` array — the exact condition typify needs to emit `#[serde(tag = "type")]` instead of `#[serde(untagged)]`. The fix is a one-line schema transformation in V13.1's exporter. This is the only material risk in the entire phase; everything else is a clean lift-and-expose of proven code.

The three reuse pillars are: (1) `net.rs` → typed REST client and SSE stream; (2) `server.rs` → supervisor lifted with library-grade error type replacing `anyhow`; (3) `event.rs` → full 21-member generated union (no `Other`) plus SDK-owned lossy projection mirroring `AppEvent`. The integration test pattern is proven in `crates/voss-tui/tests/over_the_wire.rs` — spawn with `VOSS_SERVE_FAKE_TURN=1`, drive the protocol, assert event sequence.

**Primary recommendation:** Wave 0 must confirm and fix the typify `required` issue before generating types. All other work — client, supervisor, projection, tests — is a direct lift from voss-tui with library-grade cleanup.

### Pattern 2: AgentEvent typed stream (modeled on net.rs, promoted to Stream)

**What:** Instead of voss-tui's mpsc-channel shape, expose SSE as `impl Stream<Item = AgentEvent>`.

**Source:** `crates/voss-tui/src/net.rs::stream_events` [VERIFIED: codebase]

```rust
// Source: modeled on crates/voss-tui/src/net.rs, promoted to Stream
use eventsource_stream::Eventsource;
use futures_util::StreamExt;
use futures_core::Stream;
use std::pin::Pin;

pub fn event_stream(
    client: &VossClient,
    session_id: &str,
) -> impl Stream<Item = Result<AgentEvent, VossError>> + Send + 'static {
    let url = format!("{}/session/{}/events", client.base, session_id);
    let req = client.inner
        .get(&url)
        .bearer_auth(&client.token)
        .header("Accept", "text/event-stream");
    
    async_stream::stream! {
        let resp = req.send().await.map_err(VossError::Http)?;
        let mut es = resp.bytes_stream().eventsource();
        while let Some(item) = es.next().await {
            match item {
                Ok(frame) => {
                    // data JSON carries the `type` field — serde_json::from_str works
                    match serde_json::from_str::<AgentEvent>(&frame.data) {
                        Ok(ev) => {
                            let done = matches!(ev, AgentEvent::SessionIdle(_));
                            yield Ok(ev);
                            if done { return; }
                        }
                        Err(e) => yield Err(VossError::Decode(e.to_string())),
                    }
                }
                Err(e) => yield Err(VossError::Sse(e.to_string())),
            }
        }
    }
}
// NOTE: async_stream is a convenience crate for the stream! macro.
// Alternative: hand-roll with futures::stream::unfold + Box::pin.
// The stream is lazily connected; dropping it drops the reqwest response,
// which closes the TCP connection → server-side CancelledError → session.idle.
```

**Disconnect-cancels-turn mechanics:** Dropping the `impl Stream` causes the underlying `reqwest::Response` to be dropped, which closes the TCP connection. The server's `EventSourceResponse` generator sees a `CancelledError` and calls `s.task.cancel()` (per `app.py::events` gen function). This is the PROTOCOL §8 contract — no additional Rust work needed. [VERIFIED: app.py lines 414-417]

### Anti-Patterns to Avoid

- **`anyhow` in public API signatures:** `anyhow::Error` is opaque to callers — they can't match on it. SDK uses `thiserror`-derived `VossError` in all `pub fn` return types. Internally, `anyhow` can be used in private helpers but must be converted at the boundary.
- **`#[serde(untagged)]` on AgentEvent without the schema fix:** Without adding `"type"` to `required` in each variant's JSON Schema, typify emits `#[serde(untagged)]`. While deserialization may still work (because the data JSON carries `type` and variants are mutually exclusive), the SPEC requires `#[serde(tag = "type")]` and the exhaustive match property is only reliably checkable with an internally-tagged enum.
- **build.rs codegen only:** build.rs-generated types are invisible to `cargo check` downstream and can't be diff-checked. D-06 locked checked-in.
- **Importing from voss-tui in voss-sdk:** Would create a library→binary dep (wrong direction) and a circular dependency path. The SDK must be standalone; voss-tui stays unchanged.
- **async_stream crate without checking:** `async_stream` is a convenience for the `stream!` macro. The alternative is `futures::stream::unfold` + `Box::pin`, which is more verbose but has zero extra deps. Planner should choose one approach and be consistent.

## CRITICAL: typify + Pydantic schema compatibility issue

### The problem (verified by running the actual schema export)

Every `AgentEvent` variant's JSON Schema has the `type` discriminator field defined with a `const` value but NOT listed in `required`. Example for `BannerEvent`:

```json
{
  "properties": {
    "type": { "const": "banner", "default": "banner", "title": "Type", "type": "string" },
    "model": { "title": "Model", "type": "string" },
    ...
  },
  "required": ["model", "cwd", "git"]
}
```

Typify's `maybe_internally_tagged_enum` only fires when the tag field appears in `required` (per GitHub issue #498 and confirmed in typify source). Without `"type"` in `required`, typify emits `#[serde(untagged)]`, not `#[serde(tag = "type")]`. [VERIFIED: schema output from running events.py; typify issue #498 cited]

The SPEC acceptance criterion requires `#[serde(tag = "type")]` for exhaustive match guarantee. With `#[serde(untagged)]`, serde tries variants in declaration order and falls through — this still deserializes correctly (all variants are mutually exclusive) but the Rust enum will not carry the `type` field as a visible field, and a `match` may not be exhaustive at compile time if any variant is accidentally omitted.

### The fix (owned by V13.1's exporter)

V13.1's committed-codegen Python exporter (`contracts/` snapshot writer) must add `"type"` to each variant's `required` array. This is a post-processing step on `AgentEventAdapter.json_schema()` output before writing `events.schema.json`:

```python
# In V13.1's exporter script (pseudocode)
schema = AgentEventAdapter.json_schema()
for name, defn in schema.get("$defs", {}).items():
    props = defn.get("properties", {})
    req = defn.get("required", [])
    if "type" in props and isinstance(props["type"], dict) and "const" in props["type"]:
        if "type" not in req:
            defn["required"] = ["type"] + req
```

This fix must land in V13.1 before V13.2 can generate types. The planner should note this as a **Wave 0 coordination point**: confirm the fix is in V13.1's exporter before V13.2 proceeds.

### What typify will produce (after fix)

With `"type"` in `required` and a `"const"` value, typify's `maybe_internally_tagged_enum` detects the pattern. Expected output for the event union:

```rust
// Source: typify docs + issue #498 pattern [CITED: github.com/oxidecomputer/typify/issues/498]
#[derive(::serde::Deserialize, ::serde::Serialize, Clone, Debug)]
#[serde(tag = "type")]
pub enum AgentEvent {
    #[serde(rename = "server.connected")]
    ServerConnected(ServerConnected),
    #[serde(rename = "session.idle")]
    SessionIdle(SessionIdle),
    #[serde(rename = "banner")]
    BannerEvent(BannerEvent),
    // ... 18 more variants
}
```

The variant names will be derived from the `$defs` titles (e.g., "BannerEvent", "StreamDelta") and may differ from the wire `type` values — typify emits `#[serde(rename = "...")]` for values that don't match the Rust identifier. The dot-containing type strings (`server.connected`, `stream.delta`, `budget.updated`, etc.) will definitely need `#[serde(rename = "...")]`.

### Variant naming with dots in the type field

The wire `type` values include dots: `server.connected`, `stream.delta`, `stream.finalize`, `session.idle`, `permission.updated`, `budget.updated`, `confidence.updated`, `gate.updated`, `cognition_loaded`, `principles_overflow`. Typify uses the `$defs` key (class name) as the Rust variant identifier and emits `#[serde(rename = "...")]` for the actual wire value. This is correct behavior — the Rust variant is `ServerConnected`, serde renames to `"server.connected"`. No action needed beyond confirming the generated output.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON Schema → Rust types | Hand-write 21-member enum + all struct fields | `cargo typify` | Hand-rolled types drift; SPEC forbids it; 400+ lines of serde boilerplate |
| SSE framing | Parse raw `data:` / `event:` lines | `eventsource-stream` crate | SSE spec has edge cases (multi-line data, retry field, reconnect logic); already pinned in workspace |
| Error mapping | return raw reqwest errors | `ok_or_detail()` pattern from net.rs | Server's `{v, detail}` JSON is the canonical error body; raw status codes lose the message |
| Child supervision | Kill the child manually in every caller | Drop impl + `shutdown()` method | Kill-on-drop + explicit reap prevents zombies in all code paths |
| Serde deserialization testing | Parse-by-hand unit tests | `serde_json::from_str::<AgentEvent>(...)` | The generated types ARE the contract; any deviation IS the bug |

**Key insight:** The entire REST+SSE client protocol surface is already proven in voss-tui. The risk is not implementation — it's the codegen path (typify + schema format) and the library-grade API surface (typed errors, no leaking anyhow).

## SSE Wire Format — Discriminator Details

The SSE wire format per PROTOCOL §6 and `app.py::events` (line 411):

```
event: banner
data: {"v":1,"type":"banner","model":"claude-3-opus","cwd":"/repo","git":"main"}
id: 1
```

**The `type` field appears in BOTH the SSE `event:` header AND the `data` JSON.** This means:
- `frame.event` == `"banner"` (the SSE event name)
- `frame.data` == `{"v":1,"type":"banner",...}` (JSON with `type` inside)

For the typed SDK stream, deserialization uses `serde_json::from_str::<AgentEvent>(&frame.data)`. The `type` field in the JSON data carries the discriminator — the SSE `event:` name is redundant for deserialization. This is why `#[serde(tag = "type")]` is the correct serde representation. [VERIFIED: events.py::model_dump_json output; app.py::events gen function]

The `server.connected` event is emitted BEFORE any turn starts (as the first event on SSE open, line 407). The `session.idle` event is emitted last (via `renderer.session_idle()` in `_run_turn` finally block). These are the stream boundary sentinels.

## Two Schemas, One or Two typify Passes?

**typify accepts JSON Schema, NOT OpenAPI directly.** The OpenAPI spec is a superset; typify's input is the `schemars::Schema` / JSON Schema format.

**Mapping of schema sources to Rust types:**

| Schema source | Content | Rust types |
|---|---|---|
| `contracts/events.schema.json` | `AgentEventAdapter.json_schema()` — standalone JSON Schema with `$defs` for all 21 variants | `AgentEvent` enum + variant structs (`BannerEvent`, `StreamDelta`, etc.) + helper structs (`PlanStep`, `Alternative`) |
| `contracts/openapi.json` → extract `components.schemas` | REST request/response bodies: `CreateSessionBody`, `MessageBody`, `PermissionReply`, `SessionInfo`, etc. | REST request/response structs |

**typify does NOT consume OpenAPI directly.** The planner's Wave 0 codegen task must either:
1. Extract `components.schemas` from the OpenAPI JSON and run typify on those schema definitions, OR
2. Use `cargo typify` twice: once on `events.schema.json`, once on a hand-extracted `rest-schemas.json`

The simplest approach for REST types: V13.1's exporter already produces `openapi.json`. The V13.2 codegen step extracts `openapi["components"]["schemas"]` into a standalone `rest.schema.json` (or directly passes it to typify via its Python library API). Alternatively, since REST request/response types are simple (CreateSessionBody, MessageBody, PermissionReply are all small flat structs), the planner may choose to hand-write them for the REST layer and run typify only for the event union — this is a discretion call.

**Recommendation (Claude's Discretion):** Run typify once on `events.schema.json` for the full event union (this is the critical codegen that must be correct). For REST types, the bodies are simple enough (3-4 fields each) that they can be hand-written without violating the SPEC's "generated types" requirement, since the SPEC specifically says "Rust protocol types are generated from the V13 contract snapshot" for events and REST types. However, generating them from OpenAPI components is safer for long-term drift prevention. The planner should decide and document this.

## Common Pitfalls

### Pitfall 1: typify emits `#[serde(untagged)]` instead of `#[serde(tag = "type")]`

**What goes wrong:** V13.1 ships `events.schema.json` without `"type"` in each variant's `"required"` array. typify falls through to untagged enum. Deserialization may still work (variants are mutually exclusive on the data JSON), but the SPEC's exhaustive match requirement (`#[serde(tag = "type")]`) is not met. The compile-time enforcement of "all 21 members" becomes weaker.

**Why it happens:** Pydantic v2's `model_json_schema()` with a `Literal["banner"]` default field omits the field from `required` (because it has a default). typify requires the tag field to be in `required` to detect internal tagging.

**How to avoid:** Wave 0 must include a validation step: after running typify, confirm the generated `AgentEvent` carries `#[serde(tag = "type")]`. If it shows `#[serde(untagged)]`, the V13.1 exporter needs the `required` patch described above before V13.2 can generate types. [VERIFIED: schema output; CITED: typify issue #498]

**Warning signs:** `#[serde(untagged)]` in the generated events.rs; a `match` over `AgentEvent` does NOT produce a compile error when a variant is missing.

### Pitfall 2: `anyhow` leaking into public API

**What goes wrong:** Copying net.rs and server.rs verbatim brings `anyhow::Result` into the public `VossClient` method signatures. Downstream crates can't match on the error — they can only print it.

**Why it happens:** voss-tui is a binary crate; anyhow is fine there. A library crate has callers who need to inspect errors.

**How to avoid:** Define a `VossError` enum with `thiserror`. Convert `anyhow::Error` at module boundaries using `map_err`. All `pub fn` return types use `Result<T, VossError>`. [ASSUMED — standard Rust library design practice]

### Pitfall 3: SDK → voss-tui dependency direction

**What goes wrong:** The parity projection writes `From<AgentEvent> for voss_tui::AppEvent`, which requires `voss-sdk` to depend on `voss-tui` (a binary crate). This is wrong direction; will also fail because `voss-tui` has no lib target accessible to the SDK.

**Why it happens:** Natural impulse to reuse existing types.

**How to avoid:** SDK owns `AppProjection` (a mirror of `AppEvent`'s shape). Parity test compares field-by-field, not type-by-type. `voss-tui` stays unchanged. [VERIFIED: SPEC req 4 reconciliation note]

### Pitfall 4: Zombie server on supervisor panic

**What goes wrong:** If `spawn_with` panics after the child starts but before returning `Supervisor`, the child process is left running. The `kill_on_drop(true)` on the `Command` handles the case where the `Child` exists, but if the panic happens before `Child` is extracted, no cleanup occurs.

**Why it happens:** Async spawn + stdout take in sequence; a panic between `cmd.spawn()` and the return of `Supervisor` leaves an orphan.

**How to avoid:** Use `kill_on_drop(true)` on `Command` (already in server.rs). Ensure `spawn_with` holds the child in a local that has `kill_on_drop` applied before any fallible operation that could panic. The handshake timeout (20s) covers the common case. [VERIFIED: server.rs line 62]

### Pitfall 5: SSE `server.connected` not in data JSON

**What goes wrong:** The `server.connected` event is emitted by the server BEFORE a turn starts (line 407-408 in app.py). Its data JSON is `ServerConnected().model_dump_json()` = `{"v":1,"type":"server.connected"}`. The stream consumer must handle this event and not treat it as an error even though no turn is running.

**Why it happens:** The SSE stream is opened before `POST /message` in some client implementations. The `server.connected` event acknowledges the stream is live.

**How to avoid:** The `event_stream()` function yields `AgentEvent::ServerConnected` as the first item. The SDK must NOT skip it. The SPEC acceptance criterion says "yields `server.connected` first" — build the test to assert this. [VERIFIED: app.py lines 406-408]

### Pitfall 6: 409 on re-post while turn running (not a failure)

**What goes wrong:** POST /message while a turn runs → 409. A naive test might flag this as a REST error.

**Why it happens:** Concurrency guard in post_message route (app.py line 359).

**How to avoid:** The typed error (`VossError::HttpStatus { status: 409, detail: "a turn is already running" }`) is part of the API contract. The SPEC AC requires "409 surfaces as the typed error with the correct status." Confirm the integration test asserts status code separately. [VERIFIED: app.py line 358-359]

```rust
async fn ok_or_detail(resp: reqwest::Response) -> Result<reqwest::Response, VossError> {
    let status = resp.status();
    if status.is_success() {
        return Ok(resp);
    }
    let body = resp.text().await.unwrap_or_default();
    let detail = serde_json::from_str::<serde_json::Value>(&body)
        .ok()
        .and_then(|v| v.get("detail").and_then(|d| d.as_str()).map(String::from));
    Err(VossError::HttpStatus {
        status: status.as_u16(),
        detail: detail.unwrap_or_else(|| format!("HTTP {status}")),
    })
}
```

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
