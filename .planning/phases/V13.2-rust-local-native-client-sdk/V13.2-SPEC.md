# Phase V13.2: Rust Local / Native Client SDK — Specification

**Created:** 2026-06-07
**Ambiguity score:** 0.144 (gate: ≤ 0.20)
**Requirements:** 7 locked

## Goal

Ship a reusable Rust client library crate (`voss-sdk`, name finalized in discuss-phase) for the Voss local protocol: a **generated, full-fidelity event union + REST types** (serde tagged enum, sourced from the V13 committed contract snapshot), a **typed REST client** and **typed SSE event client**, **handshake + bearer auth helpers**, a **`voss serve` child supervisor**, and **protocol-exposed session/cost/doctor readers**. The crate builds and passes integration tests against a real `voss serve`. The existing `voss-tui` binary is left unmodified this phase; crates.io publish is out of scope.

## Background

Grounded in the current repo (scouted 2026-06-07):

- **The server + protocol exist and are LOCKED.** `voss serve` (`cli.py:3722`) → `serve.py:run_server` binds `127.0.0.1:0`, prints a one-line `{"v":1,"port":...,"token":...}` handshake to stdout, supervises via getppid-poll + stdin-EOF heartbeat, runs uvicorn. Endpoints (`server/app.py`): `POST/GET /session`, `GET /sessions/saved`, `GET/DELETE /session/:id`, `POST /session/:id/message`, `POST /session/:id/abort`, `GET /session/:id/cost`, `POST /session/:id/permission`, `GET /session/:id/events` (SSE), `GET /doctor`. Auth = `Authorization: Bearer <token>` on every request, 401 at middleware otherwise.
- **The event union is the SSE contract.** `server/events.py` defines the 21-member `AgentEvent` discriminated union (discriminator `type`) + `EventEnvelope` + `PROTOCOL_VERSION = 1`. `app.py:_force_event_schema` forces `EventEnvelope` into OpenAPI components with the explicit intent "so codegen emits a tagged Rust enum."
- **A Rust client surface already exists — but inside a *binary* crate, hand-written and lossy.** `crates/voss-tui` (a `bin`) carries:
  - `server.rs` — `ServerHandle` / `spawn_server` / `spawn_server_with`: spawns `voss serve` (stdin piped as heartbeat), parses the `{port,token}` handshake, kill-on-drop + explicit shutdown. The "local server supervisor."
  - `net.rs` — `HttpClient`: REST commands + SSE consumer via `reqwest` `bytes_stream()` + `eventsource-stream`, auto-attaching `bearer_auth(token)` and the loopback base URL; non-2xx → error carrying the server's `{detail}`.
  - `event.rs` — `AppEvent`: a **hand-maintained, lossy, forward-compatible** enum decoded by `from_wire` (unknown `type` → `AppEvent::Other`, missing fields default). It is a UI projection, NOT the full protocol union.
  - `sessions.rs` / `doctor.rs` — protocol-exposed readers (`/sessions/saved`, `/doctor`).
- **No reusable Rust SDK library exists.** `voss-tui` has no `lib` target; its client modules are not a published/consumable API. `voss-auth` handles **provider** credential resolution (anthropic/codex/keychain) — distinct from the **server handshake bearer token** the SDK needs. `voss-bridge` is the subprocess JSON-RPC framing transport — a different channel from the REST/SSE server this SDK targets.
- **No audit or session-tree endpoint exists yet** — audit is V9, session-tree is V4. The only session/cost surface today is `GET /session`, `GET /sessions/saved`, `GET /session/:id`, `GET /session/:id/cost`, `GET /doctor`.
- **Hard upstream dependency:** V13 (foundation) produces the committed `openapi.json` + event-union schema this SDK's types are generated from, plus the CI drift gate. V13 must ship before V13.2 builds.
- **Sibling precedent:** V13.1 (TypeScript) locked the parallel boundary set — generated-not-handwritten, publish deferred, deep readers gated on V4/V9, real-server integration tests with the `VOSS_SERVE_FAKE_TURN` seam.

What does NOT exist yet: the `voss-sdk` library crate, the generated Rust contract/event types, a reusable typed REST/SSE client, handshake+bearer auth helpers exposed as a library API, and a Rust integration test suite against a launched `voss serve`. Those are this phase's deliverables.

## Requirements

1. **Generated typed contract**: Rust protocol types are generated from the V13 contract snapshot, not hand-authored.
   - Current: No reusable Rust contract types exist; the only Rust event model is `voss-tui`'s hand-written lossy `AppEvent`. The contract lives in Python (`events.py`, FastAPI routes) + (after V13) committed `openapi.json` + event-union schema.
   - Target: A codegen step emits Rust types for the REST request/response bodies and the full `AgentEvent` tagged union, sourced from V13's committed snapshot. The event union is a serde-tagged enum discriminated on `type` (`#[serde(tag = "type")]`), with all 21 members and their fields typed.
   - Acceptance: Regenerating types from the committed V13 snapshot produces the checked-in Rust types byte-identically (deterministic); no event/route type in the SDK is hand-maintained; a `match` over the event enum is exhaustive across all 21 members (compile error if a member is unhandled).

2. **Typed REST client**: Every protocol endpoint is callable with typed inputs/outputs.
   - Current: `voss-tui`'s `HttpClient` covers the calls but is private to the binary crate and not a reusable API.
   - Target: A library REST client covering create/list/get/delete session, `sessions/saved`, post message, abort, cost, permission reply, and doctor — each typed from the generated contract, auto-attaching `Authorization: Bearer <token>` and the loopback base URL. Non-2xx responses surface as a typed error carrying the HTTP status + the server's `{v, detail}`.
   - Acceptance: Against a real `voss serve`, the client creates a session, posts a message (202), reads cost, and deletes the session (204); a 401 (bad token) and a 409 (message while a turn runs) each surface as the typed error with the correct status.

3. **Typed SSE event client**: The event stream is consumed as a typed stream of the generated union.
   - Current: `voss-tui`'s SSE consumer decodes into the lossy `AppEvent` and forwards over an mpsc channel; not reusable, not full-fidelity.
   - Target: A library client that opens `GET /session/:id/events`, yields typed `AgentEvent` values (each frame's `data` parsed + deserialized into the tagged union), exposed as an async `Stream`. Handles the `server.connected` open and `session.idle` turn-end; dropping/closing the consumer cancels the in-flight turn (disconnect-cancels, per PROTOCOL §8).
   - Acceptance: Driving a `VOSS_SERVE_FAKE_TURN` (or real) turn, the client yields `server.connected` first and `session.idle` last, with `plan` / `stream.delta` / `final` typed in between; closing the consumer mid-turn ends the stream without a panic or leaked task.

4. **Full-fidelity union + lossy projection (voss-tui untouched)**: The SDK is the canonical full event union and provides a documented lossy projection mirroring `voss-tui`'s `AppEvent` shape, without modifying `voss-tui`.
   - Current: `voss-tui::AppEvent` is the only Rust event view and is lossy (`Other` catch-all); it lives in the binary crate.
   - Target: The generated 21-member union is the SDK's canonical event type. The SDK additionally ships a lossy/simplified projection (an SDK-owned view type or conversion) that reproduces the information content of `voss-tui`'s `AppEvent`, demonstrating the full union can produce the existing UI view. **`voss-tui`'s source is not edited in this phase** — its eventual adoption of the SDK union (and removal of the duplicate decoder) is a deferred dedupe, explicitly out of scope here.
   - Acceptance: The SDK exposes the full generated union as its primary event type; a projection function/type maps every `AppEvent` variant's payload from the corresponding generated member (covered by a unit test asserting parity for each `AppEvent` variant); `git diff` shows zero changes under `crates/voss-tui/`.

5. **Handshake + bearer auth helpers**: The SDK owns the local-server auth handshake, distinct from provider credentials.
   - Current: The `{port, token}` handshake parse + `bearer_auth` attach live inside `voss-tui` (`server.rs` + `net.rs`); `voss-auth` only resolves LLM-provider credentials.
   - Target: SDK helpers that (a) parse the one-line `{"v":1,"port":...,"token":...}` handshake line into a typed value, and (b) produce a client pre-bound to the loopback base URL + bearer token, attaching `Authorization: Bearer <token>` to every request including the SSE GET. No reuse of `voss-auth`'s provider-credential paths (different concern); `voss-bridge`'s JSON-RPC transport is not used (REST/SSE only).
   - Acceptance: Given a captured handshake line, the helper yields the correct base URL + token; a request built by the SDK carries the `Authorization: Bearer <token>` header (asserted in an integration test where a correct token succeeds and a wrong token yields 401).

6. **`voss serve` child supervisor**: A library API spawns and supervises a `voss serve` child with no orphan on drop.
   - Current: `voss-tui::server::ServerHandle` does this but is private to the binary crate.
   - Target: A reusable supervisor that spawns `voss serve --port 0` (stdin piped so the server's EOF-supervision heartbeat works), parses the handshake, returns a client bound to that port+token, and kills + reaps the child on drop and on explicit shutdown (no zombie). Interpreter selection honors `VOSS_PYTHON` then repo `.venv/bin/python` then `python3`, matching the existing behavior.
   - Acceptance: The supervisor starts a server, returns a working client, runs a turn end-to-end, and on drop/shutdown leaves no orphan `voss serve` process (verified by PID/process check); a wrong interpreter path surfaces as a typed spawn error rather than a hang.

7. **Build + integration test harness**: The crate builds in the workspace and is tested against a real server.
   - Current: No `voss-sdk` crate, no Rust SDK tests.
   - Target: `voss-sdk` is a workspace member that builds with `cargo build`, and ships an integration test suite exercising REST + SSE + permission + supervisor against a real `voss serve` (using the `VOSS_SERVE_FAKE_TURN` seam where no provider creds exist), runnable in CI.
   - Acceptance: `cargo build -p voss-sdk` and `cargo test -p voss-sdk` are green; the integration suite covers requirements 2, 3, 5, and 6 against a launched `voss serve`; codegen determinism (req 1) and the lossy-projection parity (req 4) are covered by tests in the suite.

## Boundaries

**In scope:**
- New `voss-sdk` library crate (workspace member; final name decided in discuss-phase)
- Generated Rust contract types + full 21-member `AgentEvent` serde-tagged enum (from V13 snapshot)
- Typed REST client (all current endpoints)
- Typed SSE event client (async `Stream` of the tagged union)
- Permission-reply helper (typed choice → `POST /session/:id/permission`)
- Handshake + bearer auth helpers (local-server token, not provider creds)
- `voss serve` child supervisor (spawn, handshake, kill-on-drop)
- Lossy projection mirroring `voss-tui`'s `AppEvent` shape (SDK-owned, parity-tested)
- Protocol-exposed readers (`sessions/saved`, `cost`, `doctor`)
- Build + integration test harness against a real `voss serve`

**Out of scope:**
- crates.io publish / release pipeline — separate release decision (mirrors V13.1's deferred npm publish)
- Editing `crates/voss-tui` — left unmodified this phase; its adoption of the SDK union + removal of the duplicate decoder is a deferred dedupe
- Audit & session-tree reader APIs — no such endpoint exists yet (audit = V9, session-tree = V4); readers limited to current `/session` + `/cost` + `/doctor` surface; deep readers land when V4/V9 freeze
- Reimplementing Python orchestration semantics (EM/board/runtime) in Rust — explicit non-goal; Rust is a local *client*
- Reuse of `voss-auth` provider-credential resolution or `voss-bridge` JSON-RPC transport — different concerns; SDK targets REST/SSE + the handshake bearer only
- Rust codegen tool choice (`typify` / `progenitor` / hand-rolled from the schema), HTTP/SSE crate selection, and crate/module layout — HOW, decided in discuss-phase
- Any change to `PROTOCOL.md` v1 or the server endpoints — V13.2 consumes the locked contract
- Native IDE / Tauri integrations built on the SDK — downstream consumers, not this phase

## Constraints

- **Hard dependency on V13** — types are generated from V13's committed `openapi.json` + event-union schema; V13.2 cannot build its generated types until V13 ships that snapshot. (V13.2 may be SPEC'd/planned ahead, but execution blocks on V13.)
- **Generated, not hand-written** — hand-authoring event/route types is forbidden; it reintroduces the drift V13's gate prevents. The generated union must be exhaustive (no `Other` catch-all in the canonical SDK type — unlike `voss-tui`'s lossy `AppEvent`).
- **Serde tagged enum** — the event union deserializes via `#[serde(tag = "type")]` matching the Python discriminator, per the `_force_event_schema` intent.
- **Integration tests run against a real `voss serve`** — the only faithful test of the handshake, Bearer auth, SSE framing, and supervision; use the `VOSS_SERVE_FAKE_TURN` seam to avoid provider credentials in CI.
- **Loopback + Bearer only** — no TLS (loopback), `Authorization: Bearer <token>` on every request including the SSE GET.
- **`voss-tui` byte-stable** — no source change under `crates/voss-tui/` in this phase.
- **Workspace integration** — `voss-sdk` is added to the root `Cargo.toml` `members`; reuse workspace dependency pins (`tokio`, `reqwest` rustls, `serde`, `eventsource-stream`, etc.) rather than introducing divergent versions.
- Crate name (`voss-sdk` vs alternative), module format/feature flags, Rust codegen tool, and HTTP/SSE crate choices are finalized in discuss-phase.

- [ ] REST client creates → messages (202) → reads cost → deletes (204) a session against a real `voss serve`
- [ ] 401 (bad token) and 409 (message during a running turn) surface as typed errors with correct HTTP status
- [ ] SSE client yields `server.connected` first and `session.idle` last, with typed events between; closing the consumer mid-turn ends the stream cleanly (no panic / leaked task)
- [ ] Permission helper unblocks a gate with an allow reply and blocks with a deny reply (verified via the next `tool` event state)
- [ ] Handshake helper parses `{v,port,token}` and the built client carries `Authorization: Bearer <token>`; correct token succeeds, wrong token → 401
- [ ] Supervisor spawns `voss serve`, returns a working client, and leaves no orphan process on drop/shutdown (PID/process-verified); bad interpreter path → typed spawn error
- [ ] Lossy projection reproduces every `voss-tui` `AppEvent` variant's payload from the generated union (parity unit test)
- [ ] `git diff` shows zero changes under `crates/voss-tui/`
- [ ] `cargo build -p voss-sdk` and `cargo test -p voss-sdk` are green; integration suite is green against a launched `voss serve` in CI
