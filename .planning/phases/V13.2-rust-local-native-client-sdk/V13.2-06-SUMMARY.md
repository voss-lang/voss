## Root Cause of the Handshake Timeout (handoff blocker)

The handoff suspected subprocess cwd/env. It was neither. Evidence-led diagnosis:

1. Confirmed the spawned child was the correct venv interpreter, in state `RN` (burning CPU), never emitting the handshake line — not crashed, not blocked on I/O.
2. `sample` showed a deep repeating import stack; `faulthandler.dump_traceback_later` snapshots pinned it to:
   - `litellm/litellm_core_utils/get_model_cost_map.py → fetch_remote_model_cost_map → httpx.get(...)` — **litellm does a blocking network fetch of its remote model-cost map at import time** (~12s, hang-prone).
   - and `import litellm` cold-compiling its enormous module tree: `-X importtime` measured **~44s cold** (`litellm.utils` ~20s cumulative), ~7.8s warm.
3. litellm is pulled in eagerly at `voss.cli` import via `voss.harness → voss_runtime → providers.litellm_provider → import litellm`, so the cost is paid before `run_server` even prints the handshake.

Net: warm + remote fetch = ~20s, cold = ~44s — both exceed the old 20s handshake timeout. My first manual run happened to hit warm caches + fast network (3s), which masked it. The tests failed because `spawn_with` never set the litellm local-map flag, so every spawn paid the network fetch.

**Fix:** force litellm's bundled local cost map on spawn (kills the network call — a correctness win; a local SDK server should not phone home at boot) and widen the handshake budget for cold compiles.

## Deviations from Plan

- Plan assumed `cargo typify` alone yields a tagged enum once `"type"` is in `required`. typify still emits `#[serde(untagged)]`, so a post-processing script (`generate_sdk_events.py`) converts it; the drift gate proves the committed file regenerates byte-identically through that pipeline.
- The plan did not anticipate a server-startup defect. Fixing the integration "full suite green" criterion required a supervisor change (litellm flag + timeout), beyond the plan's `files_modified` list.
