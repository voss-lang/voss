# Phase M8: Project Memory (MEM-01) — Specification

**Created:** 2026-05-14
**Ambiguity score:** 0.17 (gate: ≤ 0.20)
**Requirements:** 7 locked

## Goal

Give the Voss harness a persistent project-memory layer with two surfaces: (a) a root `VOSS.md` file that consolidates and replaces the M2-generated `.voss/architecture.md`, and (b) a cross-session recall store, backed by the existing `voss_runtime/memory/` primitives, that indexes prior turn content, written decisions, user-confirmed conventions, and per-run changed-file ledgers.

## Background

The runtime already has working `EpisodicMemory`, `SemanticMemory` (optional `chromadb` extra), and `WorkingMemory` dataclasses under `voss_runtime/memory/`. M2 (Project Cognition) shipped the `.voss/` filesystem layout — `project.json`, `architecture.md`, `decisions/`, `sessions/`, plus a per-run ledger in COG-08. Sessions today rehydrate to one `EpisodicMemory` per session (`voss/harness/session.py:166`); there is no bridge across sessions, no harness-level use of `SemanticMemory`, no human-curated root project guide (`VOSS.md`), and no `/recall`/`/forget`/`/memory`/`/save` slash commands. This phase wires the runtime primitives into the harness as a persistent agent-facing memory layer and replaces M2's auto-generated `architecture.md` with a richer root-level `VOSS.md`.

## Requirements

1. **VOSS.md loader + injection**: A root `VOSS.md` file is read at session start and injected into the harness system context.
   - Current: No `VOSS.md` file convention exists. `.voss/architecture.md` (auto-generated by M2 / COG-02) is not surfaced as system context.
   - Target: On every `voss chat` / `voss do` / `voss resume`, the harness reads `./VOSS.md` (root of project) and injects it into the system context under a labeled section before the first user turn.
   - Acceptance: Dump the system context for a fresh session; assert a `# VOSS.md` (or equivalently labeled) section is present with bytes matching the on-disk file. Absence of `VOSS.md` on disk degrades silently (no section, no error).

2. **Architecture.md → VOSS.md migration**: Existing `.voss/architecture.md` content is folded into the new `VOSS.md` without data loss, and the COG-02 write path is rewired to update `VOSS.md` going forward.
   - Current: COG-02 writes `.voss/architecture.md` via a one-shot `fs_write` from the analyze skill (`voss/harness/skills/analyze.py`, `voss/harness/cognition.py:608`). No `VOSS.md` exists.
   - Target: On the first M8 session in a repo where `.voss/architecture.md` exists and `VOSS.md` does not, the harness creates `VOSS.md` containing the prior `architecture.md` content under a machine-generated section header (e.g. `## Architecture (auto-generated)`); the original file is archived to `.voss/archive/architecture-YYYY-MM-DD.md`. Subsequent COG-02-style analyze runs write into the machine section of `VOSS.md` (preserving any human-edited sections) rather than to `.voss/architecture.md`.
   - Acceptance: (a) `sha256(.voss/archive/architecture-YYYY-MM-DD.md) == sha256(pre-migration .voss/architecture.md)`. (b) `VOSS.md` contains the full pre-migration content reachable via a deterministic section anchor. (c) Re-running the analyze skill updates the machine section of `VOSS.md` and leaves human-edited sections untouched (verified by diff against a fixture where a human section was added between runs).

3. **Cross-session recall store**: A persistent store under `.voss/memory/` indexes four source types and is searchable across sessions.
   - Current: No cross-session memory store exists. Episodic memory is per-session only; semantic memory is unused at the harness layer.
   - Target: Sources indexed — (i) prior-session turn content (assistant + user), (ii) `.voss/decisions/*.md`, (iii) extracted user-confirmed conventions (see Req 4), (iv) per-run changed-file ledgers from COG-08. Store uses `EpisodicMemory` for raw turn capture and `SemanticMemory` (chroma) for indexed retrieval when the `voss[search]` extra is installed; falls back to keyword/grep recall when `chromadb` is unavailable.
   - Acceptance: Fixture corpus of 5 sessions seeded with known content under each of the four source types; recall queries with known-answer ground truth must return the expected entry in the top 3 results ≥ 80% of the time when chroma is installed, and ≥ 60% of the time on the keyword fallback path. Source type is tagged on each returned hit.

4. **Conventions extraction with user confirmation**: At session end, the agent surfaces candidate user-style conventions; user explicitly accepts which to persist.
   - Current: No extraction pass exists. Decisions are mirrored to `.voss/decisions/` mechanically (recorder.py:136) but no inference of preferences happens.
   - Target: On clean session exit (any non-error termination of `voss chat` / `voss do`), the harness runs an extraction prompt against the session turn history and surfaces a list of candidate conventions (each with confidence and supporting evidence). The user picks which to persist; selected entries land under `.voss/memory/conventions/YYYY-MM-DD-<slug>.md`. Each session may persist zero or more entries; zero candidates skips the prompt entirely.
   - Acceptance: Driving a scripted session that contains a clear convention signal (e.g. user corrects style twice in the same direction) surfaces at least one candidate at session end. Declining all candidates writes nothing. Accepting one writes one file under `.voss/memory/conventions/` whose contents include both the convention statement and the evidence snippet.

5. **Slash command surface**: Four slash commands ship in this phase, wired into the existing harness slash registry (`voss/harness/slash.py`).
   - Current: Existing slash commands in `voss/harness/slash.py` cover session-level operations. No memory commands exist.
   - Target: `/recall <query>` returns top-N (default 5) tagged hits with source type, session id, and excerpt; `/forget <pattern>` deletes matching entries across all four source types with a confirmation prompt; `/memory` prints a markdown summary of current store contents (counts per source type, recent entries, store size on disk); `/save <note>` appends a free-form note to a manual-notes source under `.voss/memory/notes/YYYY-MM-DD-<slug>.md`.
   - Acceptance: Each of the four commands is registered, has a `--help` line, and is exercised by an integration test that asserts the expected file or stdout effect. `/forget` requires an explicit confirmation flag in non-interactive mode.

6. **Hard size cap + vacuum**: The memory store has a documented size cap and a CLI command to reclaim space.
   - Current: No size cap or vacuum mechanism exists at the harness layer. `SemanticMemory` persists indefinitely under `chroma/`.
   - Target: Default cap of 100 MB across `.voss/memory/` (configurable via `.voss/config.yml`). When writes would exceed the cap, the oldest entries (by source-tagged timestamp) are evicted until the store fits under cap. A `voss memory vacuum` command is added that compacts the chroma collection, deletes entries marked tombstoned by `/forget`, and reports bytes reclaimed.
   - Acceptance: Seed the store to 110% of cap; the next write triggers eviction; post-write size ≤ cap; oldest source-tagged entry no longer present. `voss memory vacuum` on a store with `/forget`-tombstoned entries reports a nonzero bytes-reclaimed value and the tombstoned entries are physically gone.

7. **Runtime-primitive reuse**: All persistence and retrieval routes through `voss_runtime/memory/` types, not a parallel store.
   - Current: `EpisodicMemory`, `SemanticMemory`, `WorkingMemory` exist in `voss_runtime/memory/` but the harness uses only the in-session `EpisodicMemory`. No harness code instantiates `SemanticMemory`.
   - Target: Cross-session recall (Req 3), conventions persistence (Req 4), and recall slash commands (Req 5) instantiate and operate on `voss_runtime/memory/` types. No new memory dataclass is added at the harness layer; harness code in `voss/harness/memory*.py` is a thin orchestrator over the runtime types plus filesystem layout.
   - Acceptance: Grep `voss/harness/` for memory class definitions; the count of `class .*Memory` definitions equals zero outside `voss_runtime/memory/`. The recall pipeline test (Req 3) instantiates `voss_runtime.memory.SemanticMemory` (verified by patching its `__init__` to record a call).

## Boundaries

**In scope:**
- Root `VOSS.md` file convention, loader, and system-context injection.
- Migration of `.voss/architecture.md` content into `VOSS.md` with archive of the original.
- Rewire of COG-02 analyze-skill write path to update the machine section of `VOSS.md`.
- Cross-session recall store under `.voss/memory/` covering four source types: turns, decisions, conventions, changed-file ledgers.
- Auto-triggered conventions extraction at session end with user-confirmation prompt.
- Four new slash commands: `/recall`, `/forget`, `/memory`, `/save`.
- `voss memory vacuum` CLI subcommand.
- Hard size cap + eviction policy.
- Semantic backing via existing `SemanticMemory` (chroma) when `voss[search]` extra installed; keyword fallback otherwise.
- Configuration surface in `.voss/config.yml` for cap value and extraction toggle (`--no-extract` equivalent).

**Out of scope:**
- Cross-project memory sharing — privacy + scope concerns, addressed only if real-user demand surfaces post-M8.
- Cloud-backed memory store — local-first by mandate; cloud sync is a separate v0.2+ conversation.
- TUI browser for memory store — `/memory` ships only as a markdown stdout dump in this phase; richer interactive browse lands with M9 TUI Shell.
- Multi-agent memory partitioning — relevant only after M10 multi-agent-in-chat exposes sub-agent panels.
- Promoting `voss[search]` to a default install — install footprint and dependency weight stay opt-in; M8 must work usefully (with keyword fallback) without it.
- New runtime memory primitives — M8 is harness wiring only; if the runtime needs new memory types they belong in a `voss_runtime` follow-up phase.
- Migrating pre-M8 session files to a new on-disk format — backward compat with the existing session JSON layout is mandatory.
- Encryption of the memory store — handled (if needed) by OS-level filesystem encryption; M8 does not implement crypto.

## Constraints

- All persistence and retrieval must route through `voss_runtime/memory/` types (see Req 7). No parallel memory class hierarchy at the harness layer.
- Must run on the v0.1 harness surface (`voss chat`, `voss do`, `voss resume`, `voss sessions`). No dependency on M9 TUI Shell — but reserve slash-command names so M9 panels can render the same data without rename.
- Must preserve backward compatibility with pre-M8 session JSON files: `voss resume <pre-M8-id>` must rehydrate without crash; absence of memory-store entries for that session is acceptable.
- Semantic backing is a soft dependency on `chromadb`. When the `voss[search]` extra is not installed, recall must degrade to a keyword/grep path with a documented hit-rate floor (Req 3 acceptance) — not raise an import error.
- Hard cap on memory store size (default 100 MB) must be enforced on write — the harness must never silently grow the store past cap.
- `VOSS.md` lookup is root-only in this phase. Hierarchical resolution (root + per-directory) is explicitly deferred; do not add it speculatively.
- Convention extraction adds an LLM call at session end. Must be skippable via configuration; default-on but skippable. Extraction must not block session exit by more than a documented timeout (TBD by planner — set in CONTEXT.md).
- No new entries may be written to `.voss/memory/` while a session is mid-run on the same repo (concurrent-session safety) — file locking or last-writer-wins policy decided in CONTEXT.md, but cannot corrupt the store.

## Acceptance Criteria

- [ ] `VOSS.md` is loaded and injected into system context on every supported session entry point (`voss chat`, `voss do`, `voss resume`). Absence of file degrades silently.
- [ ] In a repo containing a pre-M8 `.voss/architecture.md`, first M8 session creates `VOSS.md` containing the prior content under a deterministic section header and archives the original to `.voss/archive/architecture-YYYY-MM-DD.md` byte-identical to the pre-migration file.
- [ ] Cross-session recall fixture eval hits ≥ 80% top-3 with `voss[search]` installed and ≥ 60% top-3 on keyword fallback over a 5-session corpus seeded with known-answer queries.
- [ ] Each returned recall hit is tagged with its source type (turn / decision / convention / ledger) and a referenceable identifier (session id, decision filename, etc.).
- [ ] Conventions extraction surfaces ≥ 1 candidate on a scripted session containing a clear convention signal; declining all candidates writes nothing; accepting writes one file under `.voss/memory/conventions/`.
- [ ] `/recall`, `/forget`, `/memory`, `/save` are registered slash commands with `--help` and each has an integration test asserting effect.
- [ ] `/forget` requires explicit confirmation flag in non-interactive mode.
- [ ] Seeding the store to 110% of cap triggers eviction on the next write; post-write store size ≤ cap.
- [ ] `voss memory vacuum` on a store with `/forget`-tombstoned entries reports nonzero bytes reclaimed and removes the tombstoned entries from the on-disk store.
- [ ] No `class .*Memory` definitions exist in `voss/harness/` (grep-verified); all memory types come from `voss_runtime/memory/`.
- [ ] `voss resume <pre-M8-session-id>` rehydrates without crash on a v0.1 session JSON fixture.
- [ ] Recall degrades to keyword path without raising `ImportError` when `chromadb` is uninstalled.
