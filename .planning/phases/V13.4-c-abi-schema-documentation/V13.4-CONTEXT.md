<domain>
## Phase Boundary

Ship ONE prose reference document that tells a native / C embedder how to consume Voss's local protocol as a stable, versioned JSON-schema/ABI contract. **Doc-only — zero code.** No client library, no C headers, no `cbindgen`, no FFI/cgo. The doc points at V13.1's committed `contracts/` snapshot + `.planning/PROTOCOL.md` as the authoritative contract, gives a JSON→native type/field reference, states the stability tier of the native contract surface, and explicitly defers C headers + FFI until a concrete native embedder demands them. Lowest-priority SDK sub-phase; "may stay deferred indefinitely" is itself an acceptable outcome.

</domain>

<decisions>
## Implementation Decisions

### Doc identity
- **D-01:** Doc path = **`docs/native-embedding.md`** (the SPEC working name — locked). Single markdown file under `docs/`.

### References-resolve verification (VSDK-C-06 — the one hard gate)
- **D-02:** Verification = a **small committed shell/script check** (working location `docs/check-native-embedding-refs.sh` or a Make/CI target — exact mechanism is research/planning discretion) that enumerates every path the doc cites (`.planning/PROTOCOL.md`, `contracts/openapi.json`, `contracts/events.schema.json`, `docs/sdk.md`, `docs/ORCHESTRATION_LAYERS.md`) and exits non-zero on any missing file, AND greps `docs/ORCHESTRATION_LAYERS.md` for the named V13 tier terms. **Authorship proceeds NOW**; this check only PASSES once V13.1 ships `contracts/` and V13 ships the tier taxonomy. A script (over a manual checklist) is preferred so it can run in CI after the upstream phases land. (Final script-vs-checklist choice = discretion.)

### Reference-table approach
- **D-03:** The JSON→native reference table names `events.schema.json` as the **authoritative member source** — it does NOT hardcode the 21-member union list as the contract (that would drift). Table documents the `type` discriminator, the `v` protocol-version field, required-vs-optional handling, and JSON scalar/composite → native-reading guidance. Table granularity = discretion within these contents.

### Dependency posture
- **D-04:** **Soft dependency on V13.1 + V13; one hard gate.** The doc references V13.1's to-be-committed `contracts/` snapshot and V13's to-be-authored tier taxonomy by path/name. Author now against the to-be-committed artifacts; only VSDK-C-06's refs-resolve check hard-blocks, and only its run-and-pass step (not authorship) waits for V13.1 + V13. V13.4 **cross-links** `docs/sdk.md` — it does NOT do V13's `sdk.md` "no service, no client" reconcile.

### Claude's Discretion
- Exact verification mechanism (script vs checklist) + its filename, reference-table granularity/row structure, doc section ordering, and prose depth — all research/planning discretion within the SPEC acceptance criteria.

</decisions>
