# Phase V13.4: C ABI / Schema Documentation — Specification

**Created:** 2026-06-07
**Ambiguity score:** 0.144 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

Ship a single **prose reference document** that tells a native / C embedder how to consume Voss's local protocol as a stable, versioned JSON-schema/ABI contract — pointing at V13.1's committed `contracts/` snapshot + `PROTOCOL.md`, providing a JSON→C type/field reference, stating the stability tier of the native contract surface, and explicitly deferring C headers + an FFI/cgo shim until a concrete native embedder demands them. This phase ships **no code** — no client library, no C headers, no FFI.

## Background

Grounded in the current repo (scouted 2026-06-07):

- **This is the doc-only sub-phase of the SDK track.** Unlike V13.1 (TypeScript), V13.2 (Rust), and V13.3 (Go) — which all ship client libraries generated off the V13.1 contract snapshot — V13.4 is "NOT a full SDK" (ROADMAP §V13.4): stable JSON-schema/ABI documentation for native embedders, with C headers and FFI explicitly "much later" / "only if a concrete native embedder demands it." It is the lowest-priority sub-phase and "may stay deferred indefinitely."
- **No native/C surface exists.** `grep` for `cbindgen` / `#[no_mangle]` / `extern "C"` across `crates/*/src` returns nothing. There is no FFI, no C header, no native embedding doc.
- **The wire contract is LOCKED and is the "ABI."** `.planning/PROTOCOL.md` v1 is the authoritative wire contract (loopback REST/SSE, `{v,port,token}` handshake, Bearer auth, the `EventEnvelope`/`AgentEvent` 21-member discriminated union, permission/abort/error model). For a native embedder there is no C struct ABI today — the stable interface IS the JSON protocol over loopback. "ABI documentation" here means documenting that JSON contract as the thing a native embedder builds against.
- **The schema artifacts are owned by V13.1, not V13.** The committed `contracts/openapi.json` + `contracts/events.schema.json` + exporter + CI drift gate are V13.1 deliverables (per V13.1 CONTEXT D-10); V13.2–.4 reuse them. **Neither is on disk yet — V13.1 is unexecuted.** So V13.4's doc references artifacts that do not exist yet.
- **The SDK strategy docs (V13) are also unexecuted.** `docs/ORCHESTRATION_LAYERS.md` has no "SDK Surface Matrix" / five-tier taxonomy yet (grep empty), and `docs/sdk.md:240` still asserts the stale "HTTP/remote SDK. Voss is local-first by design. No service, no client." with no C/ABI section. The V13 foundation's job is to add the matrix + five tiers (`stable-now` / `experimental` / `generated-from-protocol` / `private-internal` / `deferred`) and reconcile `sdk.md`. **V13.4 does NOT do V13's `sdk.md` reconcile** — it cross-links it.
- **Sibling precedent.** V13.1/.2/.3 SPECs (2026-06-07) fixed: generated-from-the-snapshot, deep readers gated on V4/V9, publish deferred, no orchestration reimplementation, no `PROTOCOL.md` edits. V13.4 inherits the "no PROTOCOL edits / consumes the locked contract" posture but ships docs instead of a client.

What does NOT exist yet: the native-embedding reference doc, any JSON→C reference, and any statement of the native contract's stability tier. Those are this phase's deliverables. C headers, cbindgen, and an FFI/cgo shim are intentionally NOT deliverables.

## Requirements

1. **Native-embedding reference doc** (VSDK-C-01): a single prose document exists that orients a native / C embedder.
   - Current: No native-embedding doc exists; `docs/` has `sdk.md` (Python-centric, stale C-less) and `ORCHESTRATION_LAYERS.md` (no native section).
   - Target: A new markdown doc (working name `docs/native-embedding.md`; final filename decided in discuss-phase) that states, in prose, how a native/C embedder consumes Voss locally: spawn/attach `voss serve`, the loopback `127.0.0.1` + `{v,port,token}` handshake, `Authorization: Bearer <token>`, and JSON over REST/SSE as the contract. No code, no headers.
   - Acceptance: The doc file exists under `docs/`; it contains a section addressed to native/C embedders describing the loopback + handshake + Bearer + JSON-over-REST/SSE consumption model; a reader who has never seen Voss can identify, from this doc alone, the transport and auth a native embedder must implement.

2. **Contract source-of-truth pointers** (VSDK-C-02): the doc points at the authoritative schema + protocol, not a re-description.
   - Current: No doc links the wire contract for embedders; `PROTOCOL.md` and the (future) `contracts/` snapshot are not referenced from any embedder-facing page.
   - Target: The doc references, by full path, `.planning/PROTOCOL.md` (v1 wire contract) and V13.1's committed `contracts/openapi.json` + `contracts/events.schema.json` as the single sources of truth, and cross-links `docs/sdk.md` and `docs/ORCHESTRATION_LAYERS.md`. The doc must NOT duplicate or fork the schema — it points at it.
   - Acceptance: The doc cites `.planning/PROTOCOL.md`, `contracts/openapi.json`, and `contracts/events.schema.json` by path; it contains no copied/inlined JSON Schema body (a grep shows pointers, not a forked schema); the `sdk.md` and `ORCHESTRATION_LAYERS.md` cross-links are present.

3. **JSON→C type/field reference** (VSDK-C-03): a reference table maps the wire JSON to native-consumable guidance.
   - Current: No mapping from the JSON contract to how a C/native consumer should read it.
   - Target: A reference table (or tables) in the doc covering (a) the SSE event model — the `type` discriminator and that the union has a fixed member set sourced from `events.schema.json` (the doc states the member count is authoritative-from-schema rather than hardcoding a possibly-stale list), and (b) the core REST request/response bodies — mapping JSON types (string/number/bool/object/array/null) to native reading guidance (e.g. how a C embedder parses the discriminated union, which fields are required vs optional). It documents the `v` protocol-version field and the discriminator mechanism.
   - Acceptance: The doc contains a JSON→native reference table that names the `type` discriminator, documents the `v` version field, and maps each JSON scalar/composite type to native-reading guidance; the table references `events.schema.json` as the authoritative member source rather than asserting a frozen enumerated list as the contract.

4. **Stability statement** (VSDK-C-04): the doc states the stability tier of the native contract surface.
   - Current: No stability commitment for a native/C contract exists; the five-tier taxonomy itself is a V13 (unexecuted) deliverable.
   - Target: The doc states that the JSON wire contract a native embedder consumes is stable-as-`PROTOCOL.md`-v1 (changes require a `v` bump + migration note), maps it to the V13 five-tier taxonomy term for protocol-derived surfaces (`generated-from-protocol`), and states that the C-header/FFI surface is `deferred`. If the V13 taxonomy is not yet published when this doc is written, the doc names the tiers by their canonical names and cross-links the (pending) V13 definition.
   - Acceptance: The doc contains a stability section stating (a) the JSON contract is tied to `PROTOCOL.md` v1 + version-bump policy, (b) the protocol-derived surface's tier name, and (c) that C headers/FFI are `deferred`; the tier names used match the V13 five-tier set (`stable-now`/`experimental`/`generated-from-protocol`/`private-internal`/`deferred`).

5. **FFI / C-header deferral + trigger** (VSDK-C-05): the doc explicitly defers the C ergonomics and records what would activate them.
   - Current: Roadmap implies the deferral; no doc records it or its trigger.
   - Target: The doc explicitly marks C headers, `cbindgen`, an FFI/cgo shim, and high-level C ergonomics as **out of scope / deferred-until-a-concrete-native-embedder**, with the trigger condition recorded (a concrete native embedder with a real integration need appears) and a note that, until then, native embedders use the JSON-over-loopback contract directly.
   - Acceptance: The doc contains a "deferred" subsection naming C headers, cbindgen, and FFI/cgo as not-built, stating the trigger condition that would promote them, and pointing native embedders at the JSON contract in the meantime.

6. **References-resolve verification** (VSDK-C-06): the doc's cited artifacts actually exist — the one hard gate.
   - Current: The cited `contracts/*.json` do not exist yet (V13.1 unexecuted); the V13 tier taxonomy is unpublished.
   - Target: A verification step confirms every file path the doc cites resolves to a committed file (`PROTOCOL.md`, `contracts/openapi.json`, `contracts/events.schema.json`, `docs/sdk.md`, `docs/ORCHESTRATION_LAYERS.md`) and that the named V13 tier terms appear in `ORCHESTRATION_LAYERS.md`. This is the **only hard-blocking** acceptance item; doc authorship proceeds now, and this check runs/passes once V13.1 and V13 ship their artifacts.
   - Acceptance: A check (script or manual checklist in the phase) enumerates the doc's cited paths and confirms each resolves to a committed file (exit/flag non-zero on any missing path); when run after V13.1 + V13 ship, it passes with zero unresolved references.

## Boundaries

**In scope:**
- A single new prose doc under `docs/` (working name `docs/native-embedding.md`) for native/C embedders
- The loopback + handshake + Bearer + JSON-over-REST/SSE consumption model, in prose
- Pointers to `PROTOCOL.md` + V13.1's `contracts/openapi.json` + `contracts/events.schema.json` (no forked schema)
- A JSON→native type/field reference table (discriminator, `v` version, required/optional, scalar/composite mapping)
- A stability statement mapping the native contract to the V13 five-tier taxonomy + a `deferred` mark for C headers/FFI
- An explicit FFI/C-header deferral subsection with the activation trigger
- A references-resolve verification (the single hard gate, post V13.1/V13)
- Cross-links to `docs/sdk.md` and `docs/ORCHESTRATION_LAYERS.md`

**Out of scope:**
- Any C headers / `cbindgen` output — deferred until a concrete native embedder demands them (roadmap-gated; none exists)
- Any FFI / cgo shim or high-level C ergonomics — same deferral
- Any client library or generated code in any language — V13.1/.2/.3 ship those; V13.4 ships docs
- The `docs/sdk.md` "no service, no client" reconcile and the SDK Surface Matrix / five-tier taxonomy authoring — those are **V13's** deliverables; V13.4 cross-links them, does not author them
- Producing or copying the `contracts/` snapshot — owned by V13.1; V13.4 references it
- Any change to `PROTOCOL.md` v1 or the server endpoints — V13.4 documents the locked contract, never edits it
- Deep audit / session-tree / replay contract documentation — gated on V4/V9; only protocol-exposed surfaces are documented

## Constraints

- **Doc-only — zero code.** No source files, no headers, no FFI, no build changes. The deliverable is a markdown document (plus an optional verification checklist/script for VSDK-C-06).
- **Soft dependency on V13.1 + V13; one hard gate.** The doc references V13.1's `contracts/` snapshot and V13's tier taxonomy by path/name. Authorship proceeds now against the to-be-committed artifacts; the **only** hard-blocking item is the VSDK-C-06 references-resolve check, which can only pass once V13.1 (`contracts/`) and V13 (tier taxonomy in `ORCHESTRATION_LAYERS.md`) ship.
- **No schema fork.** The doc points at `events.schema.json` / `openapi.json` as the authoritative source; it must not inline or duplicate the schema body (that would create the drift V13.1's gate exists to prevent). It also must not hardcode the event-union member set as the contract — it names the schema as authoritative.
- **No `PROTOCOL.md` edits / no `sdk.md` reconcile.** V13.4 consumes the locked protocol and cross-links `sdk.md`; the `sdk.md` "no service, no client" fix and the matrix/tier authoring belong to V13.
- **`deferred` is a first-class state.** Per roadmap, the whole phase "may stay deferred indefinitely"; the doc itself records C headers/FFI as `deferred` with a trigger, and that is a complete, acceptable outcome — not an unfinished one.
- Doc filename/path, the exact reference-table granularity, and the verification mechanism (script vs checklist) are finalized in discuss-phase.
