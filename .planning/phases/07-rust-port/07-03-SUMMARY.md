## Refresh URL redirection mechanism

Chose a **builder method** on `AnthropicOAuthProvider`:
`with_token_url_override(impl Into<String>)`. Stored as
`Option<String>` and threaded into `voss_auth::refresh::refresh_anthropic`'s
existing `token_url_override` parameter (already added in R2 for the
`refresh.rs` wiremock tests).

Reasons:
- Keeps the default constructor (`AnthropicOAuthProvider::new(creds)`)
  ergonomic for production use â€” no extra `None` argument.
- Mirrors the existing `with_base_url(...)` builder, which is the only other
  test-seam knob on the provider.
- Avoids leaking a constructor flag that production code would always pass
  `None` for.

## Insta snapshot review

Generated by `INSTA_UPDATE=always cargo test -p voss-providers --test
anthropic_request_snapshot`. Reviewed contents at
`crates/voss-providers/tests/snapshots/anthropic_request_snapshot__anthropic_request_body_with_response_schema.snap`.

Key invariants the snapshot now locks:
- `system` is a `[{type:text, text:<preamble>}, {type:text, text:<harness>}]`
  list with the verbatim Claude Code preamble first.
- `tool_choice = {type:"tool", name:"submit_response"}` when
  `response_schema` is set.
- `tools` carries one entry â€” `{name:"submit_response", description:..., input_schema:<schema>}`.
- `model` is the alias-resolved name (`claude-sonnet-4-5` here).
- `max_tokens`, `temperature`, `messages` carry the request values.

Two non-issues to note in the snapshot:
- `temperature: 0.20000000298023224` â€” f32 â†’ f64 widening artifact when
  serializing `f32` through `serde_json`. Python emits `0.2` from a Python
  float (f64). Wire format is still numerically equivalent for Anthropic
  (numbers are floats), and the snapshot test is for *Rust-side* drift
  detection, not Python parity. Acceptable.
- `messages` order matches Python: harness system messages live in the
  `system` block list, NOT in `messages`. Only `user`/`assistant` turns
  appear in `messages`.

Subsequent runs of the snapshot test compare against this fixture without
the env override; any drift fails the test.

## Schema parity strictness

The gate in `crates/voss-providers/tests/schema_parity.rs` checks two
properties of each schema and explicitly excludes a third:

| Check | Plan | ToolCall | Why |
|-------|------|----------|-----|
| Property-name set parity | enforced | enforced | Catches add/remove/rename. |
| Required-set parity | enforced | enforced | Catches subtle drift where a field becomes mandatory on one side only â€” would crash deserialization. |
| Description / title text | NOT enforced | NOT enforced | Pydantic `Field(description=..., title=...)` adds metadata schemars omits; this is cosmetic. |
| Numeric range constraints | NOT enforced | NOT enforced | Pydantic `Field(ge=0.0, le=1.0)` on `confidence` produces `minimum`/`maximum`; schemars without `validate(range(...))` does not. Future work: add a `schemars`-compatible range derive on `confidence`. |
| Field types | NOT enforced explicitly | NOT enforced explicitly | Type drift would break compile-time round-tripping in any downstream code that calls `serde_json::from_value::<Plan>(...)`; structural type checks are deferred to runtime fixture roundtrips in later waves. |

The required-set parity is the load-bearing check. Current required sets
(verified at test time):

- `Plan.required = {"rationale", "confidence"}`
- `ToolCall.required = {"name"}`

These match Python exactly given the `#[serde(default)]` annotations on
`steps`, `open_question`, `final_when_done` (Plan) and `args`, `why`
(ToolCall) in `crates/voss-agent/src/plan.rs`.
